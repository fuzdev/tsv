# tsv_css

> CSS parser and formatter ‚Äî drop-in for Svelte's `parseCss` (AST), near-Prettier formatting

## Architecture Position

Depends on `tsv_lang` for shared primitives (spans, doc builder, comments, embedding config). Consumed by `tsv_svelte` for `<style>` block embedding, and by `tsv_cli` / `tsv_wasm` / `tsv_ffi` for top-level CSS files.

**Sources of truth**: Svelte's `parseCss` defines the public AST shape; Prettier's `css` printer defines formatter output. Both are checked at fixture-validation time.

Standard `ast/lexer/parser/printer` crate layout ‚Äî see [root CLAUDE.md ¬ßProject Structure](../../CLAUDE.md#project-structure) and [`tsv_lang/CLAUDE.md`](../tsv_lang/CLAUDE.md) for the cross-cutting types (`DocArena`, `Span`, `EmbedContext`). The internal-AST/wire-JSON split follows the workspace convention ‚Äî see [root CLAUDE.md ¬ßAST Architecture](../../CLAUDE.md#ast-architecture-internal-ast-vs-wire-json).

## Public API

**Standalone** (top-level CSS files):

- `parse(source, arena: &'arena Bump) -> Result<CssStyleSheet<'arena>>` ‚Äî parse a full CSS file. The internal AST is bump-arena-allocated (caller-owns-`Bump`); `CssStyleSheet<'arena>` borrows from it. Matches `tsv_ts`/`tsv_svelte`'s signature for the shared `lang_bindings!` macro + CLI/FFI/WASM callers.
- `format(&stylesheet, source) -> String` ‚Äî format with default config; `format_in(&stylesheet, source, &DocArena)` is the same into a caller-provided doc arena (multi-file drivers reuse one via `DocArena::reset()`; `format` is the fresh-arena wrapper)
- `convert_ast_json_bytes(&stylesheet, source) -> Vec<u8>` / `convert_ast_json_string(...) -> String` ‚Äî the **sole emission path** (bytes: FFI/CLI non-pretty; string: the same bytes plus one output UTF-8 validation, for WASM `JSON.parse` / N-API): the writer (`ast/convert/write.rs`) walks the *internal* AST once and emits the wire JSON directly into a pre-sized buffer (`tsv_lang::estimated_json_capacity`), never materializing a typed public tree, fusing byte‚Üíchar offset translation into the walk (each position through a `ByteToCharMap`; identity on ASCII). Matches Svelte's `parseCss()` JSON shape, reusing the raw-source reconstruction helpers in `ast/convert/mod.rs` (`strip_css_comments`, `split_declaration_svelte_compat`, `raw_selector_name`, ‚Ä¶). Gated against the canonical `parseCss` `expected.json` and `corpus:compare:parse --multibyte-only`
- `convert_ast_json(&stylesheet, source) -> serde_json::Value` ‚Äî a thin wrapper (`serde_json::from_slice(&convert_ast_json_bytes(...))`) for the `Value` consumers (the CLI's `--pretty`, the fixture gate); not an independent conversion
- `convert_ast_json_bytes_no_locations(...)` / `convert_ast_json_string_no_locations(...)` ‚Äî the span-only variant, present only for parity with the TS/Svelte writers and the uniform `lang_bindings!` macro. `parseCss` emits **no** per-node `loc`, so the CSS wire is already offset-only ‚Äî these are **exact aliases** of the plain functions (a documented no-op, not a distinct shape). The CLI/FFI/N-API/WASM `no-locations` surface therefore excludes a CSS-specific row.

**Embedding** (used by `tsv_svelte` for `<style>` blocks):

- `parse_embedded(source, base_offset, arena: &'arena Bump) -> Result<CssStyleSheet<'arena>>` ‚Äî same parser, but span positions are shifted by `base_offset` so they index into the parent Svelte file; `arena` is the host document's `Bump`, so the embedded CSS AST shares it
- `format_embedded(&stylesheet, source, EmbedContext)` ‚Äî formats with `EmbedContext::base_indent_offset` so wrapped lines respect outer Svelte indentation. `format_embedded_in(..., &DocArena)` is the arena-sharing form the Svelte host uses (builds the CSS doc nodes into the host's doc arena rather than a fresh per-block one; still renders to an owned `String`, and the arena is not reset so the host's in-flight nodes stay valid)
- `ast::convert::write_css_node` ‚Äî emits one embedded-`<style>` CSS node's wire JSON (no `metadata`); `tsv_svelte`'s writer composes it for the `<style>` element's `children`

## Distinctives

- **No canonical CSS parser of our own** ‚Äî fixture validation parses through Svelte's `parseCss`, not a standalone CSS reference parser. Practical implication: AST _shape_ questions go to Svelte, not the CSS Syntax spec directly. _Validity_ (what to accept/reject) is a different axis: the north star is CSS-spec compliance, the near-term enforced goal is parity with `parseCss`, and where Svelte over-accepts invalid CSS the spec wins (tsv rejects). The parser currently **hard-fails** on the first invalid construct; spec-style error recovery is a committed post-v0.2 goal. See [`../../docs/conformance_svelte.md`](../../docs/conformance_svelte.md) ¬ßCSS Parser Scope & Error Model.
- **CSS escapes are positional Unicode** (`\XXXXXX` with an optional whitespace terminator), unlike `tsv_lang::escapes` (which handles JS-style quote swapping). They decode in two places by token kind: **string values** in `escapes.rs` (`decode_escape_sequences`, at parse time) and **identifiers** in `lexer/identifiers.rs` (`decode_unicode_escape`, at lex time). Identifier decoding is **lazy** ‚Äî `read_identifier` allocates a decoded `String` only when a `\` escape is actually present; the no-escape common case allocates nothing and the identifier's text is recovered as a verbatim source slice (so the lexer's out-of-band `decoded` slot and the parser's `current_decoded` are `None` for plain identifiers). `tsv_ts` has its own `lexer/escapes.rs` for the same reason; `tsv_svelte` has none and delegates.
- **An escape is opaque to every scanner, not just to the decoders.** Decoding (above) answers *what does this escape mean*; the separate, equally load-bearing question is *how far does it reach* ‚Äî and a scanner that gets it wrong reads the escaped character as structure. `escapes.rs::escape_len` is the single definition: a **literal** escape is `\` + one code point, whose payload is content (CSS Syntax 3 ¬ß4.3.4; `\` + newline is not an escape at all); a **hex** escape is `\` + up to six hex digits + **one optional whitespace terminator that belongs to the escape** (¬ß4.3.7 ‚Äî so `\41 2px` is the single ident `A2px`, not `A` then `2px`). Every scanner that walks a value, selector, or prelude for *structure* steps over an escape whole via `escape_len` ‚Äî the value cursor and its separator classifiers, the top-level and comma splitters, the whitespace-run collapser, the selector and pseudo printers, `url.rs`, and the at-rule prelude readers. Otherwise a `\` before a delimiter (space, comma, `)`, `;`) is mistaken for that delimiter, and the output either stops parsing (`50px\ ;` ‚Üí `50px\;`, the `;` now escaped) or silently changes meaning (`.a\ , .b` ‚Üí `.a\,`, one class named `a,` and the selector list gone). `trim_end_preserving_escape` / `trim_start_css` trim **CSS** whitespace only (¬ß4.2 ‚Äî five ASCII chars, deliberately *not* `str::trim`'s Unicode `White_Space`, since NBSP and U+3000 are value *content*) and hand back a literal escape's trailing payload rather than stranding its backslash. Prettier corrupts several of these shapes identically; tsv diverges because output that does not re-parse is never the defensible side (conformance_prettier.md ¬ßCSS: Values / ¬ßCSS: At-Rules).
- ‚öÝÔ∏è **The lexer's escape terminator is a different rule with a different oracle.** `lexer/identifiers.rs::decode_unicode_escape` terminates a hex escape on **Unicode**-wide whitespace, not CSS whitespace, because its oracle is `parseCss` ‚Äî Svelte's `read_identifier` matches the terminator with a JS `\s` regex. That is not the same question as the printer's trim (¬ß4.2, ASCII), and unifying the two moves the **AST**: narrowing the lexer splits one ident into two selectors and truncates an at-rule name. Lexer boundaries become wire spans; keep the two rules apart.
- **`convert` feature flag** (default-on) ‚Äî gates the `ast::convert` layer (the writer) plus the `serde_json` dep. Disabled in the `@fuzdev/tsv_format_wasm` build, which only needs to format. See [`tsv_wasm/CLAUDE.md`](../tsv_wasm/CLAUDE.md).
- **Comments live on `CssStyleSheet`**, not on nodes. Value comments (e.g., `color: /* x */ red;`) are detected by scanning source text directly in `printer/mod.rs::has_value_comments_in_decl`, since they're not stored as `Comment` entries.
- **Gap comments are printable only if the parser registers them.** The printer reconstructs selector/prelude gaps from the AST and re-emits comments via `comments_in_range` lookups, so any gap whose parse path skips with the dropping `skip_whitespace_and_comments` loses them silently (a block-comment content-loss class `swallow_audit` cannot see ‚Äî it audits `//` line-comment swallows). Use `skip_whitespace_registering_comments` in printer-reconstructed gaps; the dropping variant is safe only where the range re-emits verbatim or another mechanism recovers the comment (the declaration property‚Üícolon gap). Selector lists, `:nth-*()`/`::slotted()`/`::part()` args, the `:dir()`/`:lang()`/`::highlight()` and unrecognized functional pseudo-class selector args, and structured at-rule preludes all register today. **Combinator-boundary comments** (a comment in a gap between compounds or glued between compound members ‚Äî `div /* c */ p`, `.a/* c */.b`) also register. `build_complex_selector_doc` detects one via `complex_has_boundary_comment` and routes to `build_complex_selector_doc_with_comments`, which re-emits the selector inline with each gap comment interleaved at its authored position and the surrounding gap whitespace **normalized to a single space** ‚Äî the same rule as the other selector-comment positions (`:is()`/`:nth-*()`/`::slotted()` args), and unlike prettier, which freezes the source whitespace. A glued compound-internal comment (`.a/* c */.b`) is emitted glued (no spaces) so the compound never reads as a descendant `.a .b`. Prettier freezes ‚Üí `_prettier_divergence` (conformance_prettier.md ¬ßCSS: Comments); parseCss rejects these entirely (`css_expected_identifier`) ‚Üí also `_svelte_divergence` (conformance_svelte.md ¬ßCSS Corrections).
- **Consecutive combinators are preserved, so `RelativeSelector.selectors` can be empty.** A run of combinators with no compound between them (`+ ~ .d`, `> > .a`, glued `>>.a`) parses into one **empty-compound `RelativeSelector`** per anchorless combinator (`combinator: Some(_)`, `selectors: &[]`). tsv's internal AST deliberately diverges from `parseCss` here ‚Äî `parseCss` collapses the run (its `read_selector` never emits an empty relative selector, dropping all but the last), a lossy recovery tsv declines to keep the authored combinators for the future diagnostics layer (a `_svelte_prettier_divergence`; see [`../../docs/conformance_svelte.md`](../../docs/conformance_svelte.md) ¬ßCSS Corrections). Both selector printer paths (`build_complex_selector_doc` and the comment-bearing `build_complex_selector_doc_with_comments`) special-case an empty compound ‚Äî the combinator emits with no trailing space (the next relative selector's separator supplies it) ‚Äî so **any code walking `RelativeSelector.selectors` must not assume it is non-empty** (a `selectors[0]` index panics on these; the trailing-combinator guard in `parse_complex_selector` keeps the run from ever ending on an empty one).
- **Printer is organized by CSS spec hierarchy** (`rules.rs` ‚Üí `declarations.rs` ‚Üí `values.rs`, plus `atrules.rs` and `selectors.rs`), not by node kind. `selectors.rs` is shared between rules and at-rules, and rule and at-rule block bodies iterate their children through one shared `print_css_block_children` routine in `mod.rs` (blank-line preservation, inline trailing comments, format-ignore ‚Äî one place, no drift); the `value_normalization/` module normalizes value text to prettier's form (numbers, hex colors, whitespace). Its number/dimension normalizers return `Cow`, borrowing an already-canonical value (`10px`, `0.5rem`) unchanged, so `build_dimension_doc` emits it as a zero-allocation verbatim `source_span` ‚Äî which is why the CSS printer renders through a source-aware `TextResolver` (a bare source-only one; unlike `tsv_ts`/`tsv_svelte` it carries no interner, since it never emits `DocText::Symbol`).
