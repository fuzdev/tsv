# tsv_css

> CSS parser and formatter — drop-in for Svelte's `parseCss` (AST), near-Prettier formatting

## Architecture Position

Depends on `tsv_lang` for shared primitives (spans, doc builder, comments, embedding config). Consumed by `tsv_svelte` for `<style>` block embedding, and by `tsv_cli` / `tsv_wasm` / `tsv_ffi` for top-level CSS files.

**Sources of truth**: Svelte's `parseCss` defines the public AST shape; Prettier's `css` printer defines formatter output. Both are checked at fixture-validation time.

Standard `ast/lexer/parser/printer` crate layout — see [root CLAUDE.md §Project Structure](../../CLAUDE.md#project-structure) and [`tsv_lang/CLAUDE.md`](../tsv_lang/CLAUDE.md) for the cross-cutting types (`DocArena`, `Span`, `EmbedContext`). The internal-AST/wire-JSON split follows the workspace convention — see [root CLAUDE.md §AST Architecture](../../CLAUDE.md#ast-architecture-internal-ast-vs-wire-json).

## Public API

**Standalone** (top-level CSS files):

- `parse(source, arena: &'arena Bump) -> Result<CssStyleSheet<'arena>>` — parse a full CSS file. The internal AST is bump-arena-allocated (caller-owns-`Bump`); `CssStyleSheet<'arena>` borrows from it. Matches `tsv_ts`/`tsv_svelte`'s signature for the shared `lang_bindings!` macro + CLI/FFI/WASM callers.
- `format(&stylesheet, source) -> String` — format with default config; `format_in(&stylesheet, source, &DocArena)` is the same into a caller-provided doc arena (multi-file drivers reuse one via `DocArena::reset()`; `format` is the fresh-arena wrapper)
- `convert_ast_json_bytes(&stylesheet, source) -> Vec<u8>` / `convert_ast_json_string(...) -> String` — the **sole emission path** (bytes: FFI/CLI non-pretty; string: the same bytes plus one output UTF-8 validation, for WASM `JSON.parse` / N-API): the writer (`ast/convert/write.rs`) walks the *internal* AST once and emits the wire JSON directly into a pre-sized buffer (`tsv_lang::estimated_json_capacity`), never materializing a typed public tree, fusing byte→char offset translation into the walk (each position through a `ByteToCharMap`; identity on ASCII). Matches Svelte's `parseCss()` JSON shape, reusing the raw-source reconstruction helpers in `ast/convert/mod.rs` (`strip_css_comments`, `split_declaration_svelte_compat`, `raw_selector_name`, …). Gated against the canonical `parseCss` `expected.json` and `corpus:compare:parse --multibyte-only`
- `convert_ast_json(&stylesheet, source) -> serde_json::Value` — a thin wrapper (`serde_json::from_slice(&convert_ast_json_bytes(...))`) for the `Value` consumers (the CLI's `--pretty`, the fixture gate); not an independent conversion
- `convert_ast_json_bytes_no_locations(...)` / `convert_ast_json_string_no_locations(...)` — the span-only variant, present only for parity with the TS/Svelte writers and the uniform `lang_bindings!` macro. `parseCss` emits **no** per-node `loc`, so the CSS wire is already offset-only — these are **exact aliases** of the plain functions (a documented no-op, not a distinct shape). The CLI/FFI/N-API/WASM `no-locations` surface therefore excludes a CSS-specific row.

**Embedding** (used by `tsv_svelte` for `<style>` blocks):

- `parse_embedded(source, base_offset, arena: &'arena Bump) -> Result<CssStyleSheet<'arena>>` — same parser, but span positions are shifted by `base_offset` so they index into the parent Svelte file; `arena` is the host document's `Bump`, so the embedded CSS AST shares it
- `format_embedded(&stylesheet, source, EmbedContext)` — formats with `EmbedContext::base_indent_offset` so wrapped lines respect outer Svelte indentation. `format_embedded_in(..., &DocArena)` is the arena-sharing form the Svelte host uses (builds the CSS doc nodes into the host's doc arena rather than a fresh per-block one; still renders to an owned `String`, and the arena is not reset so the host's in-flight nodes stay valid)
- `ast::convert::write_css_node` — emits one embedded-`<style>` CSS node's wire JSON (no `metadata`); `tsv_svelte`'s writer composes it for the `<style>` element's `children`

## Distinctives

- **No canonical CSS parser of our own** — fixture validation parses through Svelte's `parseCss`, not a standalone CSS reference parser. Practical implication: AST _shape_ questions go to Svelte, not the CSS Syntax spec directly. _Validity_ (what to accept/reject) is a different axis: the north star is CSS-spec compliance, the near-term enforced goal is parity with `parseCss`, and where Svelte over-accepts invalid CSS the spec wins (tsv rejects). The parser currently **hard-fails** on the first invalid construct; spec-style error recovery is a committed post-v0.2 goal. See [`../../docs/conformance_svelte.md`](../../docs/conformance_svelte.md) §CSS Parser Scope & Error Model.
- **A declaration's value is bounded by a byte scan, not a token walk — so `parser/decl_scan.rs` is a second reader of the lexer's grammar.** The value's text is re-parsed from source by `parser/value/` regardless, so driving the lexer across it merely to find the `;`/`}` terminator (and `!important`, and whether it holds a comment) produced a token stream nobody kept; the same is true of the `identifier :` lookahead that tells a declaration from a nested rule (`is_type_selector_with_pseudo`). Both now scan bytes. **The consequence a contributor must know: `decl_scan` mirrors the rules that decide a token's *extent*** — `url(…)` opacity and the identifier-token-start test that gates it (`5url(` is a `Dimension` plus a plain paren, so its interior is *not* url content), strings, comments, escapes, and what counts as whitespace. It reuses the lexer's own `IDENT_CONTINUE_LUT` and `is_ascii_css_whitespace` rather than re-spelling them. Two properties keep it honest: it **declines** (falling back to a real token walk) on anything it does not fully model — an escape, a non-ASCII byte at content position, an unterminated string/comment — so it never has to reproduce a lexer *error*; and a **debug-only assertion re-derives its every result from the token walk**, so a lexer change that outruns it fails the test suite instead of silently corrupting a parse. Change token extents in `lexer/` and read `decl_scan` in the same breath. The scan also reports *which* token terminates the value — a `;`, a `}`, or end-of-source — and `seat_at_terminator` makes that token current by constructing it, so the terminator is never lexed either; a second debug assertion re-lexes it and checks the construction.
- **The value span the scanner hands over is already trimmed, and `parse_value_from_source` is built around that.** Real stylesheets do not put a whitespace byte at either end of a declaration value, so `locate_value` settles the common case with two byte comparisons and never walks the text — which matters because `str::trim*` is Unicode-aware (a `char` decode and a `White_Space` test per end). Only an **ASCII non-whitespace** byte at each end settles it: a non-ASCII byte cannot (a multi-byte char's leading byte says nothing about whether that char is `White_Space`), and an ASCII whitespace byte means there is something to trim; both fall through to the trimming arm. ⚠️ The test is `char::is_whitespace`, **not** `u8::is_ascii_whitespace` — the two disagree on the **vertical tab**, the same divergence that bites the selector combinators. That trimming arm is reached only by input a stylesheet does not produce, so **no fixture and no corpus diff grades it**; the `value_span` unit test beside the function is what stands under it.
- **A non-ASCII space is CSS value *content*, so every value-element-boundary trim is ASCII-only.** CSS whitespace is ASCII-only (CSS Syntax 3), so a boundary NBSP (U+00A0), em space (U+2003), … is a name code point, not a separator; trimming it drops content, and for a **string** element it deletes the *whole* element — the trim narrows the leaf's text while its span still covers the space, so `build_string_doc` extracts a span no longer beginning with a quote and emits nothing (`'x',;` → `'x';`, non-idempotent). Every such trim routes through [`whitespace::is_ascii_trim_ws`](src/whitespace.rs) (the ASCII whitespace set **incl. U+000B** — what `str::trim` strips *within* ASCII, so it is byte-identical on ASCII input; the point is what it *excludes*): `ValueParser`'s four element-boundary trims (`parse` fallback, `parse_single`, `trimmed_end`, `push_comma_segment`) and the printer's `normalize_css_whitespace` final trim. tsv keeps the glued element as one opaque token; prettier splits it into two (word + string) — a `_prettier_divergence` (`css/values/lists/comma_string_nonascii_space`). ⚠️ Asymmetry to know: `locate_value`'s own declaration-value-boundary trim (previous bullet) is still Unicode, so a **single** (non-list) value glued to a non-ASCII space (`content: <NBSP>'z'`) still drops it — closing that means trimming `locate_value` ASCII-only *and* dropping the redundant internal trims from the leaf classifiers (`parse_color`/`parse_dimension`/`extract_function_parts`), which receive pre-trimmed text by contract.
- **A `Color::Named` prints exactly like the `Identifier` it would otherwise have been, so no corpus diff can see `is_named_color` get it wrong — the generated test is what stands under it.** `format_color_from_source` renders `Named` as a verbatim `source_span` of the value, which is byte-for-byte what `build_identifier_doc` emits for the same span; the wire writer never walks the `CssValue` tree at all (it re-slices the declaration's raw source). The classification is therefore invisible to formatted output, to the wire JSON, and to every gate built on either — a lookup that wrongly rejected `red` would sail through all of them. That is a property of *today's* consumers, not of the fact: **the classification is for the linter and LSP, which read it directly, and there a wrong answer is a wrong diagnostic.** So the vocabulary sets are declared with [`keyword_set::ascii_keyword_set!`](src/keyword_set.rs), which pairs each set with an exhaustive test grading the real lookup against a plain scan of the same list. Keep that test green; it is the only thing that can fail. The macro also puts a shape pre-filter in front of the hash (every keyword is a short run of pure lowercase ASCII letters, so length, first byte, and a first-letter→length bitmap retire `var`, `solid`, `--custom-property` and friends without touching `phf`) and derives the bitmap, the length bounds, **and** a compile-time assertion of the letters-only invariant from the keyword list itself — add a keyword that breaks it and the build fails rather than the filter silently going blind. Reach for the macro for any new CSS vocabulary; `NAMED_COLORS` and `CSS_UNITS` are the two today.
- **CSS escapes are positional Unicode** (`\XXXXXX` with an optional whitespace terminator), unlike `tsv_lang::escapes` (which handles JS-style quote swapping). They decode in two places by token kind: **string values** in `escapes.rs` (`decode_escape_sequences`, at parse time) and **identifiers** in `lexer/identifiers.rs` (`decode_unicode_escape`, at lex time). Identifier decoding is **lazy** — `read_identifier` allocates a decoded `String` only when a `\` escape is actually present; the no-escape common case allocates nothing and the identifier's text is recovered as a verbatim source slice (so the lexer's out-of-band `decoded` slot and the parser's `current_decoded` are `None` for plain identifiers). `tsv_ts` has its own `lexer/escapes.rs` for the same reason; `tsv_svelte` has none and delegates.
- **`convert` feature flag** (default-on) — gates the `ast::convert` layer (the writer) plus the `serde_json` dep. Disabled in the `@fuzdev/tsv_format_wasm` build, which only needs to format. See [`tsv_wasm/CLAUDE.md`](../tsv_wasm/CLAUDE.md).
- **Comments live on `CssStyleSheet`**, not on nodes. Value comments (e.g., `color: /* x */ red;`) are detected by scanning source text directly in `printer/mod.rs::has_value_comments_in_decl`, since they're not stored as `Comment` entries.
- **Gap comments are printable only if the parser registers them.** The printer reconstructs selector/prelude gaps from the AST and re-emits comments via `comments_in_range` lookups, so any gap whose parse path skips with the dropping `skip_whitespace_and_comments` loses them silently (a block-comment content-loss class `swallow_audit` cannot see — it audits `//` line-comment swallows). Use `skip_whitespace_registering_comments` in printer-reconstructed gaps; the dropping variant is safe only where the range re-emits verbatim or another mechanism recovers the comment (the declaration property→colon gap). Selector lists, `:nth-*()`/`::slotted()`/`::part()` args, the `:dir()`/`:lang()`/`::highlight()` and unrecognized functional pseudo-class selector args, and structured at-rule preludes all register today. **Combinator-boundary comments** (a comment in a gap between compounds or glued between compound members — `div /* c */ p`, `.a/* c */.b`) also register. `build_complex_selector_doc` detects one via `complex_has_boundary_comment` and routes to `build_complex_selector_doc_with_comments`, which re-emits the selector inline with each gap comment interleaved at its authored position and the surrounding gap whitespace **normalized to a single space** — the same rule as the other selector-comment positions (`:is()`/`:nth-*()`/`::slotted()` args), and unlike prettier, which freezes the source whitespace. A glued compound-internal comment (`.a/* c */.b`) is emitted glued (no spaces) so the compound never reads as a descendant `.a .b`. Prettier freezes → `_prettier_divergence` (conformance_prettier.md §CSS: Comments); parseCss rejects these entirely (`css_expected_identifier`) → also `_svelte_divergence` (conformance_svelte.md §CSS Corrections).
- **Consecutive combinators are preserved, so `RelativeSelector.selectors` can be empty.** A run of combinators with no compound between them (`+ ~ .d`, `> > .a`, glued `>>.a`) parses into one **empty-compound `RelativeSelector`** per anchorless combinator (`combinator: Some(_)`, `selectors: &[]`). tsv's internal AST deliberately diverges from `parseCss` here — `parseCss` collapses the run (its `read_selector` never emits an empty relative selector, dropping all but the last), a lossy recovery tsv declines to keep the authored combinators for the future diagnostics layer (a `_svelte_prettier_divergence`; see [`../../docs/conformance_svelte.md`](../../docs/conformance_svelte.md) §CSS Corrections). Both selector printer paths (`build_complex_selector_doc` and the comment-bearing `build_complex_selector_doc_with_comments`) special-case an empty compound — the combinator emits with no trailing space (the next relative selector's separator supplies it) — so **any code walking `RelativeSelector.selectors` must not assume it is non-empty** (a `selectors[0]` index panics on these; the trailing-combinator guard in `parse_complex_selector` keeps the run from ever ending on an empty one).
- **Printer is organized by CSS spec hierarchy** (`rules.rs` → `declarations.rs` → `values.rs`, plus `atrules.rs` and `selectors.rs`), not by node kind. `selectors.rs` is shared between rules and at-rules, and rule and at-rule block bodies iterate their children through one shared `print_css_block_children` routine in `mod.rs` (blank-line preservation, inline trailing comments, format-ignore — one place, no drift); the `value_normalization/` module normalizes value text to prettier's form (numbers, hex colors, whitespace). Its number/dimension normalizers return `Cow`, borrowing an already-canonical value (`10px`, `0.5rem`) unchanged, so `build_dimension_doc` emits it as a zero-allocation verbatim `source_span` — which is why the CSS printer renders through a source-aware `TextResolver` (a bare source-only one; unlike `tsv_ts`/`tsv_svelte` it carries no interner, since it never emits `DocText::Symbol`).
