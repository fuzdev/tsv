# tsv_debug

> Developer tooling crate — fixtures, comparisons, corpus runs. Embeds a Deno sidecar for canonical JS tools.

Unlike `tsv_cli` (pure Rust, production binary), `tsv_debug` exists to validate and debug `tsv` against external sources of truth: Prettier (formatting), Svelte's compiler (parsing), acorn-typescript (TS AST shape). The binary `tsv_debug` powers `deno task fixtures:*`, `deno task corpus:compare:format`, `deno task bench`, and friends. For the full command catalog and workflows, see [../../CLAUDE.md §Debug Tooling](../../CLAUDE.md#debug-tooling).

## Architecture Position

Depends on `tsv_ts`, `tsv_css`, `tsv_svelte` (with `convert` features) to drive our parser/formatter, and on `tsv_cli` for shared CLI infrastructure — `tsv_cli::cli::input::{Input, InputArgs, ParserType}`, `tsv_cli::cli::format_source::{format_source, format_source_in}`, and `tsv_cli::json_utils::to_json_with_tabs` are imported directly rather than reimplemented. The crate is `tokio`-based because the Deno sidecar I/O is async.

The Deno sidecar is a **long-running subprocess pool** spawned lazily on first use and reused across all subsequent calls (`OnceCell<Vec<ActorSlot>>` in `deno/mod.rs`, round-robin dispatch over healable slots). This is orders of magnitude faster than a fresh `deno run` per call — the cost of Prettier/Svelte/acorn module load is paid once per process per `tsv_debug` invocation. The pool defaults to size 1; each sidecar is a single-threaded JS process that handles its queued requests serially (the pool is the only parallelism), so bulk workloads that issue many concurrent calls (`fixtures_validate`, `fixtures_audit`, the `fixtures_update_*` commands, the fixtures integration test) call `deno::init_bulk_pool` before first use — it sizes the pool (via `set_pool_size`/`bulk_pool_size`) and returns a task concurrency to spread the JS work across a few processes (~3 is the measured knee; more just pays extra module loads). Each sidecar runs with `PRETTIER_DEBUG=1` so prettier-plugin-svelte's verbatim-on-error fallback (whole `<script>` block echoed when the embedded formatter throws) surfaces as a tool error instead of fake-stable output. The pool is a process-global `static`, but `tokio::spawn` binds a task to whatever runtime is current at spawn time, so each sidecar's background tasks (actor loop + stdout/stderr readers) and child-process I/O are pinned to a dedicated process-lifetime runtime (`actor::sidecar_runtime`) rather than the caller's — without it a short-lived `#[tokio::test]` runtime that first touched the pool would abort those tasks on exit, and every later test would pull a now-dead actor (`DenoError::ActorShutdown`). The pool also **heals in place**: when a dispatch finds a slot's actor dead (its event loop exited after a crash/timeout), the slot respawns a fresh actor and completes that dispatch on it — safe because a request the old actor's queue never accepted provably never reached a sidecar. A request that was *delivered* and then failed gets a hard error, deliberately without retry, so a flaky oracle stays loud; likewise `run_prettier` returns a hard `DenoError::EmptyOutput` when prettier yields (semantically) empty output for non-empty input — the under-load miss — instead of `Ok("")` that downstream would treat as truth.

## Modules

- `deno/` — Sidecar plumbing: `actor.rs` owns one spawned process (`mod.rs` holds the pool + round-robin dispatch), `protocol.rs` is the JSON-lines wire format, `sidecar.ts` is the embedded TypeScript that runs inside Deno and dispatches to prettier / svelte (parse **and** `compile`) / acorn / parseCss. Pinned npm versions live in `sidecar.ts`. The `svelte-render-key` tool compiles for the server target and reduces the baked template to its browser-visible render (block-aware whitespace collapse, `<script>`/`<style>`/comments stripped) — the authoritative oracle behind the fixture render-equivalence check.
- `fixtures/` — The fixture workflow surface — `model.rs` holds the data model (`InputType`, `Fixture`, divergence-suffix rules), `discovery.rs` walks the fixtures tree, `variants.rs` discovers variant files, `mod.rs` keeps IO/parse helpers, `validation/` is the validator (`fixtures_validate`: structure rules, per-phase checks, errors, summary — including `phases/render_equivalence.rs`, which asserts every Svelte whitespace variant *renders* like `input`, not merely that it normalizes to it), `audit_signature.rs` pins prettier's multi-pass chain.
- `test262/` — ECMAScript conformance runner: `discovery.rs` walks the test262 tree, `frontmatter.rs` parses the YAML harness metadata, `runner.rs` drives our parser (`run_test`) and grades the strict subset into the differential `Manifest` (`grade_for_manifest`, sharing the `classify` skip logic with `run_test`). Pure Rust — no Deno needed. The `test262 --emit-manifest` JSON feeds `benches/js/diagnostics/test262_compare.ts` for the tsv-vs-oxc-parser comparison.
- `cli/commands/compile_fuzz/` — the differential compile fuzzer: `anchors.rs` reads the splice offsets off tsv's Svelte parse (fragment gaps, wrappable subtrees, attribute slots, script slots, and the server-DROPPED regions — a `{:catch}` body, a `<svelte:boundary>` `pending`/`failed` snippet), `operators.rs` holds the eleven feature-level mutation operators (the cross-component donor graft, and the exotic-whitespace probe over the JS/CSS/attribute/template positions whose whitespace classes disagree), and `mod.rs` owns the generate-then-grade loop. Operators are AST/feature level rather than byte level because a compile mutant must stay oracle-**compilable** to grade anything; grading runs tsv's `compile()` first and skips the sidecar on a refusal. See [../../CLAUDE.md §Debug Tooling](../../CLAUDE.md#debug-tooling).
- `diff.rs` — Colored text / JSON diffing with line-width annotations (used by `compare`, `ast_diff`, validation failures).
- `error.rs` — `DebugError` — wraps `DenoError`, `io::Error`, `serde_json::Error`; exposes `hint()` for actionable messages.
- `cli/` — argh `TopLevel`/`Subcommand` dispatch in `mod.rs` + per-command `FromArgs` modules under `cli/commands/`. Shared three-mode input plumbing (file path / `--content` / `--stdin`) comes from `tsv_cli::cli::input::InputArgs`.
- `audit/` — Shared substrate for the corpus-walking audit commands (`gap_audit` and `blank_audit` are the consumers): `ratchet.rs` (a reusable machine-generated-snapshot ratchet — the `SnapshotKey` trait + `Ratchet`/`grade`/`render`, with new-shape and stale-entry failure and a never-pinnable class for panics), `sites.rs` (site enumeration + keying — `code_regions`/`svelte_only_regions`/`injection_sites`, `string_and_template_spans` (the blank audit's string/template-interior exclusion), and the flat `site_shape` token key), `node_edge.rs` (`node_edge_key`, the enclosing-node + child-role span-containment walker behind `gap_audit --by-node`'s emitter rollup — the coarse companion to `sites.rs`' fine token shape), `properties.rs` (per-input properties — the panic-safe ledger-clean `ledger_format`/`pristine_format`, the shared `tsv_parse_to_value`/`structurally_equivalent`/`leaf_conservation_diff` reparse-skeleton helpers, the `Utf16ToByte` wire→byte map, and `f1_check` (the parse→format→reparse→structural→leaf→idempotency fixed-point core, shared by `fuzz` and `blank_audit`)), `report.rs` (a shared finding/verdict envelope `{audit, severity, confidence, site, example, verdict, detail}` keeping each audit's own detail verbatim — `Detail::Gap` / `Detail::Blank`), and `parallel.rs` (`run_pool`, the stride-chunked worker-pool driver both injection audits run their per-file loop under — a worker panic outside the per-injection `catch_unwind` is a hard failure, not a silently-dropped tally). Ledger-touching parts compile only under the `comment_check` feature. The shared `input_invalid_*` fixture-skip check (`is_input_invalid_fixture`) lives in `cli/commands/profile.rs` beside `resolve_files`, imported by every corpus-walking command.

## Distinctives

- **Three input modes, uniform across commands.** Every content-processing command accepts a file path (parser auto-detected from extension), `--content <str> --parser <type>`, or `--stdin --parser <type>`. The shared implementation is `tsv_cli::cli::input::InputArgs` — prefer it for new commands.
- **Sidecar wire format.** `WireRequest { id, tool, content, options? }` in, `WireResponse { id, ok, output?, error? }` out, newline-delimited JSON on stdio (the sidecar also emits a diagnostic `duration_ms` the Rust client doesn't deserialize). New JS tools are added by extending `sidecar.ts`'s `dispatch` switch and exposing a typed wrapper in `deno/mod.rs`.
- **Fixtures module is not just validation.** `fixtures/` owns the data model (`InputType`, fixture discovery, expected-file naming, divergence-suffix rules) used by `fixture_init`, every `fixtures_update*` command, and `fixtures_audit` — not only the validator.
- **`tsv_debug` re-exports infrastructure for tests.** `lib.rs` exposes `deno`, `diff`, `fixtures`, `test262` publicly so integration tests in `tests/` can drive the sidecar and validation logic directly.
- **Our formatter runs in-process.** Commands that need "ours" (`compare`, `ast_diff`, fixture validation) all call `tsv_cli::cli::format_source::format_source` — the same function the production `format` command uses — so there is exactly one definition of our formatter's output across the CLI and every debug tool.
