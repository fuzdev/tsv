# Prettier Conformance

Prettier was tsv's initial guide, and the formatter still tracks it for the common case ŌĆö but tsv has its own identity and makes **intentional, cataloged choices** to diverge where they're more defensible. This document catalogs those divergences along with bugs that tsv does not replicate.

## Terminology

**Matched**: tsv produces identical output to Prettier ŌĆö the goal and the common case (measure current rates with `deno task corpus:compare:format --all --summary`).

**Unmatched**: tsv produces different output. The suffix `_prettier_divergence` marks these fixtures. This document explains WHY for each case.

## Reasons tsv Differs

Each divergence is tagged with one or more `ŌŚåreason` tokens ŌĆö single greppable
units (search `ŌŚå` for every tag, `ŌŚåprettier_bug` for one category):

- `ŌŚåspec_violation` ŌĆö Prettier emits spec-violating CSS/HTML/JS. tsv follows the spec
- `ŌŚåspec_precedence` ŌĆö Prettier's output is valid but tsv emits the spec's canonical serialized form
- `ŌŚåstable_quirk` ŌĆö Prettier preserves multiple forms without normalizing. tsv normalizes consistently
- `ŌŚåprettier_bug` ŌĆö Prettier is non-idempotent, emits invalid output, or changes meaning (e.g. strips required parens). tsv produces stable, valid, meaning-preserving output
- `ŌŚåparser_compat` ŌĆö Prettier's output breaks Svelte's parser. tsv produces Svelte-compatible output
- `ŌŚåprint_width` ŌĆö Prettier allows lines to exceed printWidth. tsv breaks to stay within the limit
- `ŌŚåtabs_only_indent` ŌĆö Prettier mixes tabs and spaces under --use-tabs. tsv uses whole tabs only
- `ŌŚåbom_stripping` ŌĆö Prettier preserves byte-order marks. tsv strips them
- `ŌŚåcomment_preservation` ŌĆö Prettier moves comments to a different syntactic position. tsv preserves comment position
- `ŌŚåcontent_preservation` ŌĆö Prettier silently drops user comments. tsv preserves all comments
- `ŌŚådesign_choice` ŌĆö Other deliberate behavior differences, with rationale in the fixture

> Most `ŌŚåcomment_preservation` and `ŌŚåcontent_preservation` divergences live in the prose-form [TypeScript: Comments](#typescript-comments) and [CSS: Comments](#css-comments) catalogs, not the tag-prefixed catalog lists ŌĆö they're the largest divergence category but don't reduce to a single bullet.

## Decision Framework

**When to match Prettier:**

- Cosmetic choices (spacing preferences, quote styles)
- Output that's valid and reasonable
- Unclear which approach is "better"

**When to differ:** any reason in [Reasons tsv Differs](#reasons-tsv-differs) above. The three cross-cutting principles ŌĆö comment position, print width, and tabs-only indentation ŌĆö are detailed below.

### Comment Position Philosophy

**A formatter should not move a comment to a different syntactic position ŌĆö unless
the move is lossless and the position carries no authorship signal.** Comment
placement is usually a deliberate authoring choice ŌĆö it communicates what the comment
refers to ŌĆö so preserving it is tsv's default and its single largest category of
divergence from Prettier (see [TypeScript: Comments](#typescript-comments)).

Prettier's comment handling is its weakest area. It routinely moves comments from
between syntactic boundaries into adjacent blocks, parens, or other positions, changing
the apparent association ŌĆö and frequently **losing information** (two comments merging
onto one end-of-line, the second `//` becoming text; or reordering them). tsv treats
comment position as semantic and preserves it wherever that distinction is real.

**Principles:**

1. **Comments between an operator and its operand stay there.** If the user wrote
   `? foo : // about bar`, the comment stays after `:`. Prettier moves it to trailing
   on `foo`, changing its association from the false branch to the true branch.
2. **Trailing comments stay trailing.** `foo // comment` keeps the comment on `foo`.
3. **Same-line block comments stay same-line.** `extends T /* c */ ?` keeps the
   comment after `T`, not moved after `?`.
4. **Both positions are valid when dual-stable.** When the user's chosen position is
   idempotent, preserve it. Don't collapse to one canonical form ŌĆö that destroys the
   distinction between "comment about X" and "comment about Y".
5. **The deciding test is information loss, not position purity.** Preserve a comment's
   position when relocating it would lose information ŌĆö the canonical case is Prettier's
   end-of-line relocation *merging* two comments onto one line (the second `//` becomes
   text) or reordering them; tsv keeps them distinct (the nameŌåÆ`=`/`:`/`?` binding
   divergences). But where relocating is **lossless** *and* the position carries no
   signal ŌĆö a same-line line comment past a *pure separator*, e.g. a list element's
   comma (`A // cŌÅÄ, B` ŌåÆ `A, // c`; the comma is structure and the comment trails the
   element either way) ŌĆö tsv trails like Prettier rather than manufacturing a divergence
   for a meaningless distinction.

**When reviewing comment-related fixes:** Default to preserving position. Match
Prettier's repositioning only when the move is lossless *and* the position carries no
authorship signal (a pure-separator trail), or when the original position is clearly
wrong (e.g., comment inside a token boundary). Otherwise ŌĆö and whenever relocating
would merge, reorder, or drop a comment ŌĆö preserve and create a `_prettier_divergence`
fixture.

### Uniform Forced-Continuation Indent

A direct corollary of comment-position preservation, and tsv's most cross-cutting
comment-layout rule. When a **line** comment forces part of a construct onto a new
line ŌĆö a `//` runs to end-of-line, so whatever the author wrote after it cannot stay
on that line ŌĆö tsv keeps the comment where it was written and drops the following
token to a continuation line **indented one level**. The continuation then reads as
part of its construct, not as a sibling statement or member.

One rule, applied at every site where a line comment splits a construct's head from
its tail:

- **Declaration and module headers** ŌĆö keywordŌåÆname, keywordŌåÆ`{`, bindingŌåÆ`from`,
  `*`ŌåÆ`as`, and every other header gap (`import // cŌÅÄ{ a } from 'm'`,
  `export // cŌÅÄconst x = 1`). See [Declaration- and module-header line-comment
  continuation indent](#comment-relocation).
- **Prefix type operators** ŌĆö the `keyof`/`typeof` operand hang
  (`type A = keyof // cŌÅÄ\t\tB`), shared via `append_keyword_value_line_comments` with
  type-parameter constraint/default values and class-property initializers. See
  [Prefix type-operator operand hang](#comment-relocation).
- **`: Type` annotations** ŌĆö the colonŌåÆtype continuation (`prop: // cŌÅÄ\tType`), via
  the shared `build_type_annotation_doc`, **uniformly for union, intersection, and
  simple types** and in **every** context: property signatures
  ([annotation_simple](../tests/fixtures/typescript/types/comments/annotation_simple_prettier_divergence/)),
  variable declarations, class properties, function parameters/return types, and
  intersection types

  plus an index signature's key-type
  ([index_signature_key_type_line_comments](../tests/fixtures/typescript/types/type_members/index_signature_key_type_line_comments_prettier_divergence/))
  and value-type
  ([index_signature_value_line_comment](../tests/fixtures/typescript/types/type_members/index_signature_value_line_comment_prettier_divergence/)).
- **Before-`:` key/binding gap** ŌĆö the complement of the colonŌåÆtype case: a line
  comment between a key/binding name (or its `?`/`!` marker) and the `:`
  (`prop // cŌÅÄ\t\t: T`) keeps the comment after the marker and indents the whole
  `: type` continuation one level, via the shared `build_marker_colon_line_continuation`.
  Uniform across index signatures
  ([index_signature_key_colon_line_comment](../tests/fixtures/typescript/types/type_members/index_signature_key_colon_line_comment_prettier_divergence/)),

  and `?`ŌåÆ`:`

  properties relocates the comment to end-of-line.
- **Index-signature bracket gaps** ŌĆö the `]`ŌåÆvalue-`:` continuation
  (`[k: T] // cŌÅÄ\t: V`). See [Index signature `]`ŌåÆvalue-`:`](#comment-relocation).

The indent is tsv's own layout choice; prettier handles each site differently ŌĆö it
relocates the comment (into braces/parens, after `from`/`as`/`;`), floats it past
`;`, keeps the continuation **flush**, or drops the comment onto its own line. So
this rule is a deliberate `_prettier_divergence` everywhere it bites: where prettier
**indents** the declarator (`const`/`let`/`var` keywordŌåÆdeclarator) it agrees, a
regular fixture; the multi-member union after `:` is a divergence (prettier drops the comment
onto its own line). The payoff is
internal consistency: every forced continuation reads the same regardless of which
construct the comment split.

### Print Width Philosophy

**Prettier treats `printWidth` as a soft target.** Lines may exceed it in various edge cases (fill algorithm boundaries, block expressions, template literals, certain constructs that "don't look good" when wrapped).

**tsv treats `printWidth` as a hard limit.** If content exceeds 100 characters, tsv breaks it when possible. This is a deliberate design choice affecting many divergences in this document:

- Block expression conditions (`{#if}`, `{#each}`, etc.) with logical operators
- Template literal interpolations
- Fill algorithm edge cases (101-char boundaries)
- Single-specifier imports
- Various "short expression" tolerances

The benefit: predictable output that respects the configured line length. The tradeoff: some constructs may break where Prettier would keep them inline.

### Tabs-Only Indentation Philosophy

**tsv renders all indentation as whole tabs and never mixes tabs with alignment spaces.**
When breaking a union type, Prettier wraps each member's type doc in
`align(2, ŌĆ”)` (`union-type.js`) to offset the 2-char `|` prefix. With
`--use-tabs`, Prettier's indentation algorithm renders that 2-column offset as
a **sub-tab alignment**: content that is further indented rounds the offset up
to a whole tab, but a closing delimiter sitting at the offset column is emitted
as `tabs + 2 spaces`. The result mixes tabs and spaces on a single indentation
level.

tsv rounds the 2-column offset up to one tab everywhere. At
`tabWidth = 2` the two are the same visual width; only the representation
differs (`Ō¤©n+1 tabsŌ¤®}` vs `Ō¤©n tabsŌ¤®┬Ę┬Ę}`). This keeps indentation
tab-width-agnostic: a reader viewing tabs at any width sees consistent
structure, whereas Prettier's 2-space offset assumes the prefix is exactly 2
columns wide. Cataloged in [Tabs-Only Alignment](#tabs-only-alignment).

**Escaped whitespace ending a prelude**: an at-rule prelude can end in an escaped whitespace (`@layer a\ ;` ŌĆö the layer is named `a `, since `\ ` is a valid escape whose escaped code point IS that space, css-syntax-3 ┬¦4.3.4/┬¦4.3.7). tsv preserves it. **Spec violations**: CSS Syntax 3 ┬¦4.3.4 specifies that an identifier immediately followed by `(` tokenizes as a `<function-token>`, not as an `<ident-token>` plus `(`. Media Queries 4 ┬¦3 explicitly notes: "Whitespace is required between a 'not', 'and', or 'or' keyword and the following '(' character, because without it that would instead parse as a `<function-token>`." Container Queries (CSS Conditional 5) use the same grammar pattern. Prettier normalizes this for `@supports` but not `@media` or `@container`.

**SCSS directive numbers**: SCSS/Sass directives (`@include`, `@mixin`, `@if`, `@for`, `@each`, `@while`, `@function`, `@return`, `@debug`) are not standard CSS. Per CSS Syntax 3 ┬¦5.4.2 an unknown at-rule's prelude is consumed as an opaque list of component values with no defined grammar, so tsv preserves it verbatim (e.g. `@include foo(.5s)` stays `.5s`, `@include baz(1.50)` stays `1.50`). Prettier keeps a hardcoded SCSS-directive list whose params it re-parses as a value AST and number-normalizes (`.5s`ŌåÆ`0.5s`, `1.50`ŌåÆ`1.5`). tsv applies number normalization only to contexts whose grammar it parses ŌĆö declaration values and `@media`/`@supports` preludes (see [CSS: Values](#css-values) "Number dot-ident"); unrecognized directive preludes stay raw. Both outputs are valid CSS (`.5` and `0.5` are the same `<number>` token); the divergence is one of scope, not correctness.

**CSS keyword case ŌĆö the canonicalize/preserve line.** CSS keywords are ASCII case-insensitive (CSS Syntax 3 ┬¦"tokenizing"). tsv adopts prettier's canonicalization set wholesale ŌĆö it **lowercases** property names, at-rule names, units, pseudo-class/element names, media-feature names, keyframe `from`/`to`, hex, and `!important`, and **preserves** values (`color: RED`), custom properties (`--MyVar`), type/element selectors (`DIV`, SVG `linearGradient`), function names (`URL(`/`RGB(`), media types (`SCREEN`), and the `not`/`only` media qualifiers ŌĆö matching prettier on both sides. tsv parses every case-insensitive keyword (so `@supports (a: b) AND (c: d)`, `@media screen AND (ŌĆ”)`, `:NTH-CHILD(2n+1)` all parse ŌĆö Class-1 correctness) but only rewrites the spelling where prettier does. **Escaped whitespace in a value**: an **escape is opaque** ŌĆö nothing inside it is structure. `\` followed by a space is a valid CSS escape whose escaped code point **is that space** (CSS Syntax 3 ┬¦4.3.4 "Check if two code points are a valid escape" ŌĆö only a newline after `\` invalidates it ŌĆö and ┬¦4.3.7 "Consume an escaped code point", whose final branch returns the code point itself). This only manifests when compact input is formatted ŌĆö both formatters preserve the block form if given it directly.

**Block multiline attrs hug**: When whitespace-sensitive elements (`<pre>`, `<textarea>`) have multiline attributes and hugged content that would exceed print width, Prettier keeps `>{content}</tag>` on the attribute line (101+ chars). tsv breaks `>` to its own line (`\n>{content}`) to respect print width while preserving whitespace semantics (no text node added since `>` immediately precedes content).

**svelte:element `this`**: prettier-plugin-svelte ignores `singleQuote` for a brace-wrapped string literal in `<svelte:element this={ŌĆ”}>`, always emitting double quotes (`this={'hello'}` ŌåÆ `this={"hello"}`), and skips escaping entirely ŌĆö so `this={'a"b'}` becomes the invalid `this={"a"b"}` and `this={'a\b'}` corrupts to a backspace. tsv delegates the literal to the normal string printer, honoring `singleQuote` and escaping like any other string. The bug is narrow ŌĆö only a *directly* brace-wrapped string literal triggers it; concatenations, template literals, and other expressions delegate to the JS printer and agree (boundary fixture [svelte_element_this](../tests/fixtures/svelte/special_elements/svelte_element_this/)). One adjacent facet diverges further: a *parenthesized* literal `this={('hello')}` collapses to the plain attribute `this="hello"` under prettier (structural rewrite), while tsv keeps `this={'hello'}` ŌĆö encoded by the fixture's `unformatted_ours_paren` + `variant_paren_collapse` pair.

**svelte:element class ws**: prettier-plugin-svelte does not collapse repeated whitespace in a `class` attribute on `<svelte:element>` (`class="a   b    c"` stays verbatim), while it still collapses on a plain element (`<div class="a   b    c">` ŌåÆ `class="a b c"`) ŌĆö and so does tsv, everywhere. The `<svelte:element>` attribute path runs off a separate code path from the normal attribute printer, like the `this` bug. tsv collapses uniformly across all elements. Both formatters keep the single-spaced form stable; the divergence shows only when collapsing multi-space input.

**Space after block element**: When a block element (`<div>`, `<p>`, ŌĆ”) is directly followed by content text with a same-line (space, no linebreak) boundary, prettier-plugin-svelte trims the text's leading whitespace (`trimTextNodeLeft`) but still emits the block-child break, stranding a **leading space** on the text line (`<div>block</div> text` ŌåÆ `>ŌÅÄ text`). It is **non-idempotent** ŌĆö a second pass trims the space, converging to the same form tsv produces. tsv takes the same trim but emits no fold/group after the block (the block's own break already supplies the separating line), so no stray space survives and the result is identical in one pass from either authoring. The `unformatted_ours_compact` variant pins tsv's one-pass normalization; `prettier_intermediate_compact` captures prettier's stray-space first pass (which converges to `input` on the next pass).

### Svelte: Inline content block-style

Design choice. tsv lays out an inline element's **wrapping content** block-style ŌĆö both tags stay intact and the content moves to its own indented line(s), collapsing to `<tag>content</tag>` when it fits, exactly like a block element. Prettier instead **dangles** the tag delimiters (`<tagŌÅÄ\t>content</tagŌÅÄ>`, pre-breaking the opening tag and dangling the closing `>`). This is uniform across text, expression, and element/component children, with and without attributes (the `>` is **attr-keyed** ŌĆö it hugs the last attribute when attributes fit and dedents to its own line when they wrap), and including table cells (which are inline-classified). Content that fits stays inline; only overflow breaks to block-style.

Content-boundary whitespace is **render-free under Svelte 5** ŌĆö start/end-of-content whitespace is trimmed at compile (`<p>foo<span> - bar</span></p>` renders `foo- bar`), so the injected block-style boundaries are render-equivalent (confirmed at corpus scale by `ast_diff --render`).

Because it is render-free, that whitespace does **not select the layout**. Wherever the content goes multiline, the rule holds without exception: it holds whether the content breaks **structurally** (a block child, a control-flow block) or by **width**, and it holds for a boundary the author *hugged* (zero whitespace) exactly as for one they spaced or broke. So the authorings of one document ŌĆö `<span>text<div>ŌĆ”`, `<span> text<div>ŌĆ”`, and the newline form ŌĆö converge on one form. Prettier lets the boundary decide, so it keeps a *different* stable form for each ([content_boundary_convergence](../tests/fixtures/svelte/components/content_boundary_convergence_prettier_divergence/) is one document with three prettier forms and one tsv form), and its space form is additionally **non-idempotent** (the space is emitted at line-start, where the next parse reads it as indentation and drops it). This is the authoring-independence property `authoring_audit` checks ŌĆö it mutates each content boundary across hug Ōåö space Ōåö newline and requires every authoring to converge.

The corollary is that whitespace *inside* the content ŌĆö a **separator between two siblings** ŌĆö is render-significant and therefore does still decide layout. Whitespace between two adjacent expression tags (`{a} {b}`) is a collapsible break: a space while the element fits, a newline once it goes block-style, so each tag takes its own line ([inline_content_spaced_tags_long](../tests/fixtures/svelte/elements/inline_content_spaced_tags_long_prettier_divergence/)); a text run between them flows as one fill and keeps them on the content line. The boundary carries no signal and must not select the layout; the separator carries one and must.

**Scope: every fragment in the language, not just an element's content.** Svelte has two families of fragments ŌĆö an element's/component's content, and **block bodies** (`{#if}` / `{:else}` / `{#each}` and its `{:else}` fallback / `{#key}` / each `{#await}` phase / `{#snippet}`) ŌĆö and their boundaries are **equally render-free** (verified against `svelte/compiler`: every fragment boundary in the language is trimmed at compile; only inter-sibling whitespace and `<pre>`/`<textarea>` are significant). The stance applies to both: a block whose body renders multiline lays out block-style ŌĆö head and tail intact, body on its own indented lines ŌĆö however the author wrote that boundary, collapsing inline when the body fits. Prettier instead lets the boundary decide, keeping a hugged body hugged (`{#if c}<div>b1</div>ŌÅÄ<div>b2</div>{/if}`) even as it breaks across lines, so it holds a different stable form for each authoring of one document.

Two corollaries specific to blocks, both following from "the boundary carries no signal":

- **Hug is all-or-nothing, across the whole construct.** A body welded to only the head, or only the tail, is not a half-expansion signal ŌĆö it converges like any other authoring ([if/hugged_boundary_convergence](../tests/fixtures/svelte/blocks/if/hugged_boundary_convergence_prettier_divergence/)). And a branch that would render inline still drops to its own line once any *sibling* branch went multiline, so `{:else}` never welds its body while `{#if}` holds its own.
- **The body-expand decision is made by width alone, never by whether the head may wrap.** Those are orthogonal ŌĆö head-wrap is a property of the head, body layout a property of the body. Keying the body on the head's wrap-ability welded the body of any block following a *breakable* sibling ([expression_tag/adjacent_if_block_long](../tests/fixtures/svelte/expression_tag/adjacent_if_block_long_prettier_divergence/) and its `{@html}` / `{@render}` siblings), where prettier's welded line only fits by breaking the body's inner call arguments. Welding is not merely the worse layout, it is **unstable**: the body then renders at the column where the over-width head ended, so its fill breaks at *every* separator, and re-formatting that output reads the newline it just injected as authored leading whitespace and settles on a different form ŌĆö an **F1 (idempotency) break**. Expanding is what makes the emitted form its own fixed point.

**The scope of that rule is layout, and layout is only at stake once the content breaks.** An inline element whose content *fits* on one line still **preserves an authored boundary space** ŌĆö `<span> text </span>` and `<span>text</span>` are both stable, matching prettier ([inline_boundary_whitespace](../tests/fixtures/svelte/elements/inline_boundary_whitespace/)). Those two are the same document (the compiler removes that space), so tsv is not authoring-*independent* here; it is deliberately authoring-*preserving*, and the character it keeps costs nothing structurally. `authoring_audit` skips these sites for that reason, and it is the one place a render-free boundary character still survives in tsv's output. Trimming it too would be a further divergence from prettier with real corpus churn, and has not been taken.

Two consequences of "the boundary carries no signal":

- **A break on one side alone is not an expansion signal.** A component expands when the author broke **both** content boundaries; a lone opening newline collapses back inline rather than half-breaking the tags ([opening_newline_only](../tests/fixtures/svelte/components/opening_newline_only_prettier_divergence/)). Prettier honors the lone break and dangles the closing delimiter to hold the content against it.
- **`svelte:*` elements are not special here.** `<slot>`, `<svelte:element>`, `<svelte:component>`, `<svelte:fragment>` and friends run the **same** layout analysis as regular elements, so they converge on the same block-style form. (They are block-kind for whitespace purposes: their content boundaries are trimmed, not preserved as a space. Prettier draws the same line from the other side ŌĆö its `isInlineElement` requires a `RegularElement`.)

**A CONTENT-boundary dangle survives in exactly one place: inside `<pre>` / `<textarea>`**, where that boundary whitespace *is* render-significant and the dangle is mandatory rather than cosmetic ŌĆö see [pre_closing_tag](../tests/fixtures/svelte/elements/pre_closing_tag/), [textarea_content_long](../tests/fixtures/svelte/elements/textarea_content_long/). Those elements never reach the shared layout analysis. An *empty* whitespace-sensitive element has no content to protect, so an inline parent hugging it lays out block-style like any other ([textarea_empty_hugged_long](../tests/fixtures/svelte/elements/textarea_empty_hugged_long_prettier_divergence/)).

**tsv does keep one dangle of its own, and it is not an exception to this rule** ŌĆö the **sibling `>` dangle** (`<a href={root}>Home</aŌÅÄ>{#each ŌĆ”}`), where an inline element immediately followed by a multiline-rendering block hands its closing `>` to the block head. See [┬¦Svelte: Blocks](#svelte-blocks). The distinction is the whole point of this section: that dangle is keyed on **inter-sibling** whitespace ŌĆö specifically its *absence* ŌĆö and inter-sibling whitespace is render-**significant** under Svelte 5 (it collapses to one space; it does not vanish). So the character selecting that layout is one the compiler *keeps*, which is exactly the license a content-boundary character lacks. Same reason `authoring_audit` reshapes inter-sibling runs (spaceŌåönewline) but *creates and destroys* content-boundary ones (hugŌåöspaceŌåönewline).

Both formatters keep the block-style form once produced (`prettier(input) == input`); the divergence is that tsv *converges every authoring* to block-style while prettier dangles a compact authoring ŌĆö so these fixtures carry `unformatted_ours_*` compact variants (tsv normalizes them to the block-style input, prettier does not) alongside `prettier_variant_*` files pinning prettier's stable dangle form (which tsv likewise normalizes to the block-style input). See also the "Inline content (text)", "Inline content (expression)", and "Wide inline content + trailing text" rows above.

### TypeScript: Template Literals

For **value-position** interpolations `${...}`, tsv follows Prettier's heuristic (`template-literal.js` `printTemplateExpression`): a `${...}` is kept **on one line** unless its *source* already spans multiple lines, or the expression would render with a newline anyway (a nested function / block body). When neither holds the expression is **atomized** ŌĆö rendered flat, unable to break ŌĆö so the interpolation stays inline even past print width, exactly as Prettier does. Only when the interpolation *does* span lines does tsv break it, and then by expression type:

- **Qualifying types** (Identifier, MemberExpression, ConditionalExpression, BinaryExpression, SequenceExpression, TSAsExpression, TSSatisfiesExpression): softline wrapping at `${`/`}` boundaries ŌĆö the group breaks when the line exceeds print width.
- **Non-qualifying types** (CallExpression, chains, ArrowFunction, etc.): no softlines at `${`/`}` ŌĆö the expression breaks internally while `${`/`}` stays hugged.

Both paths match Prettier for the common value-position case ŌĆö a bare `${prop}`, `${obj.field}`, a binary operand, a ternary consequent, all past print width with compact source, covered by the non-divergence [interpolation_boundary](../tests/fixtures/typescript/expressions/literals/template/interpolation_boundary/) and [interpolation_expression_inline_long](../tests/fixtures/typescript/expressions/literals/template/interpolation_expression_inline_long/) fixtures. A non-null-asserted member chain (`obj.a.b!`) qualifies exactly like the bare member (Prettier's `stripChainElementWrappers` peels the `!` before the member test; tsv mirrors it), covered by the non-divergence [interpolation_non_null_newline](../tests/fixtures/typescript/expressions/literals/template/interpolation_non_null_newline/) fixture. - Nested-deep chain / ternary ŌĆö [interpolation_nested_deep_long](../tests/fixtures/typescript/expressions/literals/template/interpolation_nested_deep_long_prettier_divergence/)
- Nested template ŌĆö [interpolation_nested_template](../tests/fixtures/typescript/expressions/literals/template/interpolation_nested_template_prettier_divergence/)

**Atomization**: Prettier pre-renders each template expression at `printWidth: Infinity` (`template-literal.js`); if the result is single-line it replaces the expression doc with that atomic string, so the interpolation can never break. tsv reproduces this without re-rendering: when the interpolation source carries no newline and the expression doc has no forced break (`will_break` ŌĆö a doc breaks at infinite width iff it holds a hardline / forced break), tsv strips the expression's soft breaks (`remove_lines`) so `${...}` stays flat. Simple interpolations like `${prop}` or `${obj.field}` therefore stay inline in both formatters.

**Multiline indent**: For code-generation templates with indented, multi-line-source content, tsv applies Prettier's `addAlignmentToDoc` (ceiling division to match Prettier's useTabs rounding) and **matches Prettier** ŌĆö single-level indent alignment is non-divergent, covered by the non-divergence fixture [interpolation_multiline_indent_long](../tests/fixtures/typescript/expressions/literals/template/interpolation_multiline_indent_long/). **Nested template**: When a template expression contains an array literal wrapping a long inner template, tsv breaks the array to respect print width while Prettier keeps it inline.

**Template-literal types**: tsv treats print width as a hard limit for a `${...}` inside a template literal **type** ŌĆö breaking after `${` with `}` on its own line ŌĆö whereas Prettier keeps type interpolations inline regardless of length. Unlike the value-position case above (which matches Prettier), this is a deliberate tsv choice, covered by the three `template_literal_type_*` fixtures listed above.

**No blank above a body block's `{`**: tsv drops an authored blank line in a control-flow gap before a body-leading own-line comment, **uniformly** ŌĆö `while (a)ŌÅÄŌÅÄ// cŌÅÄ{` ŌåÆ `while (a)ŌÅÄ// cŌÅÄ{`, across `if`/`while`/`for`/`do`/`else`/`try`/`catch`/`switch`/ŌĆ” ŌĆö so a body block's `{` never sits below a blank. **Type-construct gap line comments** (swallow prevention): a line comment in a type construct's delimiter/keyword/operator gap is kept where the author wrote it, the following token dropped to the next line so the `//` cannot swallow it (the shared `build_trailing_comments_hang_next` / `build_leading_comments_break_for_line`). Emitting the comment inline (the previous behavior at these sites) let the `//` run to end-of-line and absorb the following token ŌĆö non-idempotent **content loss**. Covered: indexed access [indexed_access_line_comment](../tests/fixtures/typescript/types/indexed_access_line_comment_prettier_divergence/), mapped `in`/`as` [mapped/keyword_line_comment](../tests/fixtures/typescript/types/mapped/keyword_line_comment_prettier_divergence/), import-type qualifier [import_type_qualifier_line_comment](../tests/fixtures/typescript/types/import_type_qualifier_line_comment_prettier_divergence/), and tuple rest / named / optional members [tuple/member_line_comment](../tests/fixtures/typescript/types/tuple/member_line_comment_prettier_divergence/). (`infer` lands a break too but takes the indented keywordŌåÆvalue hang rather than this flush form, so it diverges ŌĆö see the Infer keyword entry above.)

**A template-literal interpolation is authoring-dependent ŌĆö the one gap that preserves *both* authorings.** The `${`ŌåÆtype gap takes neither of the forms above wholesale; it keys on where the author put the comment, and each form is a stable fixed point:

- authored **trailing** `${` ŌåÆ stays trailing, type below it (`` `a${// cŌÅÄŌćźBŌÅÄ}` ``). Prettier expands instead, moving the comment down ŌĆö the divergence ([template_literal_interp_trailing_comment](../tests/fixtures/typescript/types/template_literal_interp_trailing_comment_prettier_divergence/), width case [template_literal_interp_trailing_comment_long](../tests/fixtures/typescript/types/template_literal_interp_trailing_comment_long_prettier_divergence/), where Prettier additionally emits a blank line after `${`). Prettier's expanded output is itself a tsv fixed point (pinned as that fixture's `variant_expanded`), so tsv never pulls the comment back up ŌĆö the two authorings hold two stable forms rather than one collapsing into the other.
- authored **own-line** ŌåÆ the interpolation **expands**, the comment keeps its line (`` `a${ŌÅÄ// cŌÅÄBŌÅÄ}` ``). This is prettier's layout, so tsv matches ŌĆö no divergence ([template_literal_interp_own_line_comment](../tests/fixtures/typescript/types/template_literal_interp_own_line_comment/)).

The two questions are independent and keyed on opposite sides of the comment: what *follows* it decides whether the type hangs at all (`comment_hangs_next`, the newline after), what *precedes* it decides whether the comment keeps its own line (the newline before). A glued single-line block still collapses inline (`` `a${/* c */ B}` ``). Both broken forms put `}` on its own line, as a width-driven break does ŌĆö a hanging comment is the *only* break mechanism in play, so a too-wide type simply breaks *inside* the already-broken interpolation instead of a second break firing on top of it.

Why authoring-dependent here, when most gaps pick one canonical form: a single form **cannot** preserve both authorings ŌĆö forcing the expansion moves a trailing-authored comment down (Prettier's choice), forcing the flush form pulls an own-line-authored one up. Both authorings render identically and each form is stable, so there is no lossless-or-idempotent reason to collapse them; the authored break is simply signal, and tsv keeps it. This is the same distinction the type-alias `=` gap already draws (`type A = /* c */ B` hugs, `type A =ŌÅÄ/* c */ŌÅÄB` keeps its own line), and it makes the divergence *minimal* ŌĆö only the trailing authoring differs from Prettier.

The `=` keeps the backtick on its line for both (`value_owns_its_comment_break`), since either layout already breaks the template itself.

**Else line comment with non-block body**: `} else // c\nexpr;` ŌåÆ Prettier keeps the comment after `else` but drops it onto its own line, indented with the body (`} else\n\t// c\n\texpr;`); tsv keeps it trailing `else` on the same line (`} else // c\n\texpr;`). Affects block consequent, comments-path, and non-block consequent cases. **Default / namespace import + export-all header comments**: the same preserve-in-place rule covers the remaining module-header shapes. Where Prettier relocates (export-all, `*`ŌåÆ`as`) tsv is free; where Prettier keeps the comment in place and flat (keyword/`type`ŌåÆbinding, `type`ŌåÆnamespace-`*`, `as`ŌåÆbinding) tsv's indent is a deliberate indent-only divergence.

**Import/export binding-or-specifiers to `from`**: A comment in the gap between an import's binding/specifiers (or a re-export's specifiers) and the `from` keyword is preserved where the user placed it; Prettier's relocation depends on the binding shape. For a **default** or **namespace** binding (no braces) a same-line block comment stays in place in both formatters (`import Foo /* c */ from 'a'` ŌĆö dual-stable), but Prettier floats a line comment past the `;` to a statement-trailing position (`import Foo from 'a'; // c`, Prettier's statement-trailing float) while tsv keeps it before `from`, indenting the `from ŌĆ”` continuation one level onto its own line. For **named specifiers** (`import { a } /* c */ from 'a'`, `export { a } /* c */ from 'a'`) Prettier relocates the comment _into_ the braces as the last specifier's trailing comment ŌĆö a block comment inline (`{ a /* c */ }`), a line comment expanding the braces multiline (`{\n\ta // c\n}`) ŌĆö while tsv keeps it after `}` with the braces inline (indenting `from ŌĆ”` when a line comment forces the break). **Import attributes header comments**: A comment in an import's attributes header ŌĆö between the source and the `with` keyword (`import x from 'a' /* c */ with {ŌĆ”}`), or between `with` and the attributes `{` (`import x from 'a' with /* c */ {ŌĆ”}`) ŌĆö is preserved where the user placed it. **Keyword ŌåÆ specifier brace comments**: A comment between the `import`/`export` keyword and the named-specifier `{` (`import /* c */ { a }`, `export /* c */ { a }`) is preserved before the brace; Prettier relocates it _into_ the braces as the first specifier's leading comment ŌĆö a block comment inline (`import { /* c */ a }`), a line comment expanding the braces multiline. tsv keeps it where the user wrote it; a line comment forces `{` onto the next line, indenting the `{ŌĆ”}` continuation one level (the uniform module-header rule below). **Declaration- and module-header line-comment continuation indent**: A _line_ comment in a declaration- or module-header gap forces the following token onto a new line, and tsv indents that continuation one level ŌĆö a statement spanning lines reads as a continuation, not a second statement.

For **module headers** tsv applies this **uniformly to every gap, with no exceptions**: keywordŌåÆ`type`, keyword/`type`ŌåÆdefault-binding, keyword/`type`ŌåÆnamespace-`*`, keyword/`type`ŌåÆ`{`, keyword/`type`ŌåÆempty-`{}` (re-export _and_ no-`from`), bare keywordŌåÆsource, export-all `export`/`type`ŌåÆ`*`, `*`ŌåÆ`from`, `*`ŌåÆ`as`, `as`ŌåÆbinding, binding/specifiersŌåÆ`from`, `from`ŌåÆsource, the `with`/attributes-`{` gaps, and `export`/`export default`ŌåÆdeclaration. Prettier's own handling varies per gap ŌĆö it relocates the comment (into the braces, after `from`/`as`/`;`), floats it past `;`, or keeps it in place and flat ŌĆö but tsv **always** indents. So where Prettier relocates, tsv's in-place indent is just one of two unrelated layouts; and where Prettier keeps the comment in place and flat (`as`ŌåÆbinding, keyword/`type`ŌåÆdefault/namespace binding, `type`ŌåÆnamespace-`*`, bare/empty/export-all `from`ŌåÆsource, no-`from` empty `{}`, `export`ŌåÆdeclaration) tsv's indent is a deliberate indent-only divergence, chosen so every continuation reads alike.

For **declaration headers** the same rule covers the keywordŌåÆname gap of `function` (incl. `async function`, `function*`, and `declare function`), `class` (incl. `abstract class`), and `enum` (incl. `const enum`), plus the keywordŌåÆdeclarator gap of `const`/`let`/`var`. Prettier keeps the comment in place and flat for the `function`/`class`/`enum` headers, so tsv's indent is an indent-only divergence there; for `const`/`let`/`var` Prettier **agrees** (it also indents the declarator), so there is no divergence ŌĆö a regular fixture.

The deliberate exclusions are constructs where the following token isn't part of the same declaration: **ASI statement keywords** (`return`, `throw`, `break`, `continue`, `yield`) ŌĆö a newline after them triggers automatic semicolon insertion, so `return // cŌÅÄexpr` is two statements, not a continuation; the **contextual declaration keywords** (`type`/`interface`/`namespace`/`module`/`declare`), whose line forms never form a single declaration ŌĆö the keyword requires its following token on the **same line** (tsc's `isDeclaration`: `nextTokenIsIdentifierOnSameLine`, and the ASI modifier rule for `declare`), so a line break demotes the keyword to a plain identifier. `interface`/`namespace`/`module`ŌåÆname then become unparseable (`interface // cŌÅÄI {}` is rejected); `type`ŌåÆname and `declare`ŌåÆkeyword **ASI-split** into two statements (`type;` then `T = ŌĆ”`; `declare;` then `const x = ŌĆ”`). tsv's parser conforms here ŌĆö see [contextual_keywords/declaration_keyword_own_line](../tests/fixtures/typescript/syntax/contextual_keywords/declaration_keyword_own_line/); and **function/class expressions** (`const a = function // cŌÅÄf() {}`), which are not declaration headers and stay flat in both formatters. Block comments and the no-comment case are byte-identical in every in-scope gap. (The `export`-prefixed forms ŌĆö `export interface // cŌÅÄI` etc. ŌĆö are out of scope: after `export` the parser is committed to a declaration, where tsc does not apply the same-line gate, so tsv's acceptance there may be spec-correct; acorn-typescript rejects them, but that is a separate acorn-vs-tsc question.)

**A synthesized paren never lands between a comment and the token it is glued to** (`ŌŚåprettier_bug`, pinned by [pure_annotation_enclosing_parens](../tests/fixtures/typescript/syntax/comments/pure_annotation_enclosing_parens_prettier_divergence/)). Some comments are *bound to the token after them* ŌĆö the whole point of `/* @__PURE__ */` is that it marks the call it precedes as side-effect-free, so rollup, esbuild and terser may drop the call when its result is unused. When the formatter adds a paren around an expression whose left edge is such a call, that paren must go **outside** the comment:

```js
const a = cond ? p : (/* @__PURE__ */ f() ?? y); // tsv ŌĆö `f()` is still pure
const a = cond ? p : /* @__PURE__ */ (f() ?? y); // prettier ŌĆö the mark is now inert
```

`f() ?? y` is not a call, so no bundler reads the relocated comment as an annotation: `f()` is silently retained. Nothing was added or dropped ŌĆö the comment just crossed a boundary that changed what it binds to. This is [the JSDoc-cast bug](#jsdoc--paren-semantics) again, and it has the same cause: prettier's leading comments are applied by its *generic* printer on top of whatever the language printer returned, parens included (`src/main/ast-to-doc.js` wrapping `src/language-js/print/index.js`), so a comment is **structurally** outside the parens for every node. Its one escape hatch is `willPrintOwnComments`, an allowlist of node types. Two differences from the cast case make this one worse: prettier is **idempotent** on its own output here, so no second pass exposes the shift (the cast's non-idempotence is what surfaced *that* bug), and the annotation has no AST node at all, so there is nothing to hand ownership to.

tsv's rule is therefore about the **comment**, not the node: a comment glued to the token after it is printed by the innermost node that token begins (`Comment::owned_by_node`, set in the parser, skipped by every gap emitter). The comment travels inside its node's doc, so a paren synthesized around any enclosing expression is outside the pair by construction ŌĆö at every paren site, present or future.

**Every glued block comment is owned, not just bundler annotations.** The position ŌĆö a block comment glued to the token after it ŌĆö is an authoring choice that binds the comment to the operand it leads, so there is no content sniff: `/* @__PURE__ */` and a plain `/* c */` bind their token identically, and no synthesized paren lands between them. An ordinary leading comment in a stripped-grouping-paren position is therefore preserved glued where prettier relocates it out ŌĆö `(/* c */ x ?? y)` stays put, prettier hoists to `/* c */ (x ?? y)` ŌĆö at every operand position: the ternary operand ([test_paren_leading_comment](../tests/fixtures/typescript/expressions/ternary/test_paren_leading_comment_prettier_divergence/)), the sequence operand ([operand_edge_comment](../tests/fixtures/typescript/expressions/sequence/operand_edge_comment_prettier_divergence/)), the unary assignment operand ([assignment_operand_leading_comment](../tests/fixtures/typescript/expressions/unary/assignment_operand_leading_comment_prettier_divergence/)), and the non-null / await / yield grouped operands ([┬¦Comment relocation](#comment-relocation)). The annotation **convention** (`@`/`#` plus `__NAME__`) is retired as an ownership gate; the only remaining content sniff is `is_jsdoc_type_cast_comment`, and it governs *cast paren-retention* (the `JsdocCast` node below), never ownership ŌĆö ownership flows to a cast the same as to any other glued comment.

**Ownership makes a comment invisible to emit-decisions, NOT to layout.** This is the rule for anyone extending the mechanism. An owned comment is skipped by the *to-emit* lookups (`comments_to_emit_in_range`) ŌĆö the owning node prints it ŌĆö so any gate deciding *who emits* must use them (`has_comments_to_emit_in_range`). But the comment is still on the page, still occupies width, and still means to prettier's rules exactly what an ordinary comment means. So any gate deciding *layout* must use the *on-page* lookups (`has_comments_on_page_in_range`), which count it. Get this backwards and the comment silently disappears from a layout decision: a unary operand loses the parens that keep the comment bound to the operand rather than the operator (`!(/* @__PURE__ */ f())` ŌåÆ `!/* @__PURE__ */ f()`), and a last argument loses the leading comment that refuses the expand-last hug. Both were real bugs. The unary wrap is decided **positionally** ŌĆö the comment sits in the operatorŌåÆoperand gap, so `build_unary_doc` sees it there and adds the parens itself; `needs_parens` deliberately does *not* (it would double-wrap), and needs no left-spine walk, which is why an instantiation-expression operand (`!(/** @type {A} */ (x)<T>)`) gets the wrap for free.

#### Comments inside a multi-word keyword

A comment can sit **inside** a keyword that spans two or more words ŌĆö `await /* c */ using`,
`export /* c */ default`, `declare /* c */ const`. tsv preserves it there; Prettier relocates it
past the keyword's last word, onto the value (`await using /* c */ y`).

Preserving is the [Comment Position Philosophy](#comment-position-philosophy) default, and three
things confirm it rather than merely permit it:

1. **The carve-out doesn't reach.** tsv trails a comment past a *pure separator* ŌĆö a list
   element's comma, which is structure. A keyword's words are not separators; `using`,
   `default`, and the kind keyword each carry meaning, so the gap before them is a position an
   author can mean.
2. **Prettier's own sibling gap preserves.** `export /* c */ const x = 1` and
   `export /* c */ function fn() {}` keep the comment after `export` in *both* formatters ŌĆö only
   `export /* c */ default` relocates. Preserving makes the whole `export`ŌåÆX family read one way;
   matching Prettier would split it on nothing but the follower.
3. **Relocation collapses a distinction.** With comments on both sides of the second word,
   Prettier lands them in one place (`await /* c2 */ using /* c3 */ b` ŌåÆ
   `await using /* c2 */ /* c3 */ b`), so before-`using` and after-`using` become
   indistinguishable. Text survives; the association does not.

`for await /* c */ (ŌĆ”)` ŌĆö the gap *after* the same class of keyword ŌĆö already preserves for the
same reason ([for_await_keyword_comment](../tests/fixtures/typescript/statements/for/for_await_keyword_comment_prettier_divergence/)),
so preserving the interior gap makes the treatment uniform.

A *line* comment in such a gap takes the [uniform forced-continuation
indent](#uniform-forced-continuation-indent) (`export // cŌÅÄ\tdefault 1;`); Prettier pulls the
value back flush. Both the authored and the relocated positions are stable under tsv.

"Uniform" is meant literally, and a multi-gap keyword is where it has to be said: a header whose
gaps break **more than once** continues at **one** level, not one per gap
([export_as_namespace_line_comment](../tests/fixtures/typescript/modules/exports/export_as_namespace_line_comment_prettier_divergence/)).
Each gap is emitted without its own indent and the whole tail is wrapped once. Indenting per gap
compounds, and the staircase is not merely deep ŌĆö the keywordŌåÆvalue gap is emitted at the header's
own level, so the keyword's last word would sit *below* the value following it. Only a three-word
keyword (`export as namespace`, the import-equals header) can show the difference.

The tail has to be *inside* that one `indent`, not a sibling after it. The distinction is invisible
for a tail that can't break ŌĆö a bare name renders the same either way, because the gap's hardline
has already written the indent the name lands on. It shows only when the tail breaks **internally**,
and the import-equals module reference is the one that can: as a sibling, `require(`'s own line
breaks resolve at the *outer* level, leaving its contents level with it and its `)` a level above
the `require(` it closes
([equals_header_line_comment](../tests/fixtures/typescript/modules/imports/equals_header_line_comment_prettier_divergence/)).

Not every multi-word keyword admits an interior comment: `declare // cŌÅÄconst x` ASI-splits into
two statements, and `await /* cŌÅÄ */ using` demotes (the comment *is* a `LineTerminator`, and
`await [no LT] using` requires none) ŌĆö see the exclusions under [Declaration- and module-header
line-comment continuation indent](#comment-relocation).

The same rule covers a keyword whose second word is a punctuator (`export /* c */ = value`) and
multi-gap headers (`export /* c1 */ as /* c2 */ namespace Foo`, the import-equals header).

It reaches past keywords, too. A **meta property** is the same shape ŌĆö `new` `.` `target` ŌĆö and both
gaps around its dot are positions an author can mean (`new /* c */.target`, `new./* c */ target`).
So is a **qualified name** (`ns` `.` `Type`), which is why one printer (`build_dotted_pair_doc`)
serves both: neither can regrow the hole independently, and through the qualified name it covers
type references, interface heritage, and import-equals module references at once. tsv keeps each
comment on its authored side, which is what prettier does for a block comment; a *line* comment
there is a divergence
([dot_gap_line_comment](../tests/fixtures/typescript/expressions/misc/meta_property/dot_gap_line_comment_prettier_divergence/),
[qualified_name_dot_gap_line_comment](../tests/fixtures/typescript/types/qualified_name_dot_gap_line_comment_prettier_divergence/)).

Those two are worth knowing for a different reason: the detector for this class ŌĆö a `d.text` literal
with an *interior* space ŌĆö **cannot see them**, because the joining literal is `"."`. The class is
"a header that concatenates fixed pieces without scanning the gaps between them"; the interior-space
grep is a proxy for it, not a definition of it, so a punctuator-joined header needs finding by hand.
Both were found that way, and both had been **dropping** every comment in both gaps.

**Preserving is not the same as diverging, and most of these gaps are neither.** Prettier's
treatment is not uniform: it relocates the `export as namespace` keyword-interior gaps, but
**preserves** the `namespace`ŌåÆname gap and four of the five import-equals gaps. Where Prettier

#### JSDoc / paren semantics

- JSDoc type cast parens (standalone TS) ŌĆö [jsdoc_type_cast_ts](../tests/fixtures/typescript/syntax/comments/jsdoc_type_cast_ts_prettier_divergence/)
- JSDoc type cast in Svelte template / directive positions ŌĆö [jsdoc_cast_template](../tests/fixtures/svelte/syntax/comments/jsdoc_cast_template_svelte_prettier_divergence/)
- ŌŚåprettier_bug ŌĆö an enclosing paren emitted *between* the cast comment and its `(`, which re-binds the cast ŌĆö [jsdoc_type_cast_enclosing_parens](../tests/fixtures/typescript/syntax/comments/jsdoc_type_cast_enclosing_parens_prettier_divergence/)

**JSDoc type cast parens.** `/** @type {T} */ (expr)` is a TypeScript type **assertion** (cast): the parentheses are required, and `/** @type {T} */ expr` (no parens) is *not* a cast. tsv therefore **preserves** the parens of a JSDoc cast everywhere ŌĆö the parser records the cast in an internal `JsdocCast` wrapper node, keyed on the exact paren extent, so `(a.b)` (cast the member) and `(a).b` (cast the base) stay distinct and nested casts keep each level. The public AST stays paren-free, matching acorn/Svelte (which carry no `ParenthesizedExpression`). tsv never *adds* parens ŌĆö only a `@type`/`@satisfies` block comment immediately before a `(` triggers preservation; a non-`@type` comment, or one not adjacent to a `(`, is stripped as ordinary grouping.

The divergence is **JS-vs-TS**, keyed on prettier's parser backend rather than on file type:

- **JS contexts** (`.js`, `.svelte` plain `<script>` ŌĆö prettier's babel parser) **preserve** the cast parens too: prettier's `is-type-cast-comment.js` + the `postprocess` `shouldKeepParenthesizedExpression` check keep a `ParenthesizedExpression` whose immediately-preceding comment is a type cast. tsv **matches** here ŌĆö see the match fixtures [jsdoc_type_cast_svelte](../tests/fixtures/typescript/syntax/comments/jsdoc_type_cast_svelte/), [jsdoc_type_cast_extent](../tests/fixtures/typescript/syntax/comments/jsdoc_type_cast_extent/), [jsdoc_type_cast_nested](../tests/fixtures/typescript/syntax/comments/jsdoc_type_cast_nested/), and [jsdoc_type_cast_keyword](../tests/fixtures/typescript/syntax/comments/jsdoc_type_cast_keyword/). Because tsv formats context-free, these JS matches also **oracle-anchor the preserved layout** that the TS divergence shares ŌĆö the wide cases ([jsdoc_cast_call_arg_long](../tests/fixtures/typescript/expressions/calls/jsdoc_cast_call_arg_long/), [jsdoc_cast_multiarg_long](../tests/fixtures/typescript/expressions/calls/jsdoc_cast_multiarg_long/), [jsdoc_cast_arg_long](../tests/fixtures/typescript/expressions/calls/jsdoc_cast_arg_long/), [jsdoc_cast_paren_long](../tests/fixtures/typescript/calls/chained/jsdoc_cast_paren_long/), [jsdoc_cast_before_eq](../tests/fixtures/typescript/declarations/variable/jsdoc_cast_before_eq/), [jsdoc_cast_paren_span](../tests/fixtures/typescript/declarations/variable/jsdoc_cast_paren_span/), [arrow_jsdoc_cast_body_long](../tests/fixtures/typescript/calls/arrow_jsdoc_cast_body_long/)) pin how the preserved parens break against prettier-plugin-svelte rather than against tsv's own output.
- **TS contexts** (`.ts`, `.svelte` `<script lang="ts">` ŌĆö prettier's oxc-ts parser) **strip** the parens: oxc-ts is cast-unaware (the preservation is gated behind `if (!isOxcTs)`), so prettier silently drops the assertion. tsv preserves ŌåÆ **divergence** (the fixtures listed above). When the inner expression needs parens anyway (e.g. an assignment in `return`), prettier keeps them in both backends, so that case stays a match.

tsv runs one context-free TypeScript formatter, so "preserve" is uniform: it matches prettier in JS contexts and diverges from prettier-TS's strip. This is the **more correct** behavior, not merely a taste call ŌĆö the parens carry meaning, and prettier's own babel path special-cases preserving them (it strips meaningless parens aggressively everywhere else). The cast paren is **opaque to layout heuristics** (expand-last etc.), mirroring acorn's `ParenthesizedExpression`, so a long cast breaks via standard arg breaking and the inner expands inside the preserved parens. An object/array literal inner **hugs** the cast parens (`({ŌĆ”})` / `([ŌĆ”])`) ŌĆö see [jsdoc_type_cast_object](../tests/fixtures/typescript/syntax/comments/jsdoc_type_cast_object/) ŌĆö while a line comment between the `(` and the inner forces a hardline so it can't swallow the inner ŌĆö see [jsdoc_type_cast_gap_comment](../tests/fixtures/typescript/syntax/comments/jsdoc_type_cast_gap_comment/). Negatives that pin the boundary: [jsdoc_type_cast_member](../tests/fixtures/typescript/syntax/comments/jsdoc_type_cast_member/) and [jsdoc_type_cast_nonadjacent](../tests/fixtures/typescript/syntax/comments/jsdoc_type_cast_nonadjacent/) (comments that are not casts), [jsdoc_type_cast_word_boundary](../tests/fixtures/typescript/syntax/comments/jsdoc_type_cast_word_boundary/) (the ASCII `\b` after `@type`/`@satisfies`, mirroring prettier's regex), and [jsdoc_type_cast_ts](../tests/fixtures/typescript/syntax/comments/jsdoc_type_cast_ts/) (never-add + no double-wrap in standalone TS).

**An enclosing paren goes OUTSIDE the cast comment** (`ŌŚåprettier_bug`, pinned by [jsdoc_type_cast_enclosing_parens](../tests/fixtures/typescript/syntax/comments/jsdoc_type_cast_enclosing_parens_prettier_divergence/)). A cast is the comment **plus** the `(` it is glued to. When the formatter adds a paren around an expression whose *left edge* is a cast ŌĆö `??` clarity parens under a ternary, the required parens of a `??` operand under `||`, a member/call base, a `new` callee, a `**` operand, a value-position sequence or assignment ŌĆö that paren must land **outside** the comment. Prettier emits it **between** the comment and the cast's own `(`:

```js
const a = cond ? x : /** @type {A} */ ((b).c ?? d); // prettier, pass 1
const a = cond ? x : /** @type {A} */ (b.c ?? d); // prettier, pass 2 (fixed point)
const a = cond ? x : (/** @type {A} */ (b).c ?? d); // tsv ŌĆö cast still on `b`
```

Prettier's pass-1 output is **not a fixed point**: reparsed, the comment is now glued to the *enclosing* paren, so that paren becomes the cast and the original `(b)` decays to a redundant grouping paren, which pass 2 strips. The fixed point it lands on has **changed the program's meaning** ŌĆö `@type` annotated `b`, and now annotates all of `b.c ?? d`, so `b` goes unchecked. The same bug is in oxfmt and biome; there is no formatter that gets it right.

tsv holds the pair together **structurally** rather than re-deriving it from text: the parser gives the `JsdocCast` node ownership of its comment (`Comment::owned_by_node`), so the comment is printed by the cast and skipped by every gap emitter ŌĆö no synthesized paren can be placed between the two, at any of the ~29 paren sites, present or future. Prettier re-decides cast-hood from text adjacency on every parse, which is exactly why its own output doesn't survive a second pass.

Note that a cast re-binding is **invisible to every existing gate**: the cast is not a node in the public AST (the writer unwraps it) and grouping parens aren't either, so both forms serialize to byte-identical wire JSON ŌĆö `ast_diff` calls them "semantically equivalent", `roundtrip_audit`'s skeleton compare cannot see it, and `corpus:compare:format`'s char-frequency SAFETY check cannot either (the characters only move). This one surfaced solely because prettier's version happens to be non-idempotent, which F1 catches. The fixture is therefore the only pin, and a dedicated audit (format ŌåÆ reparse ŌåÆ compare each cast's inner subtree, on the *internal* AST) is tracked as follow-up work.

A unary operand takes the cosmetic wrapping pair prettier also emits (`!(/** @type {A} */ (expr))`): bare, the comment reads as annotating the operator rather than the operand. Both forms are the same AST; the wrap lands outside the comment either way. The rule is **positional, not structural** ŌĆö the comment sits in the operatorŌåÆoperand gap, so `build_unary_doc` sees it there and adds the parens, which means it applies to *any* operand shape (a member chain, a `**` operand, an instantiation expression) and to a bundler annotation as well as a cast, with no node list to keep in sync. An operand that already prints **its own value-position pair** is the exception: that pair encloses the comment, so the comment-holder wrap is not added again ŌĆö `!(/* c */ x = y)` (one pair), never `!((/* c */ x = y))`. This covers every `needs_parens` operand ŌĆö an assignment, a conditional, an arrow, await/yield, a type assertion ŌĆö plus a sequence (whose own printer supplies the pair). Binary is the one operand that is *not* the exception: its plain path doesn't prepend the owned comment, so it keeps taking the comment-holder pair (a single pair ŌĆö the redundant inner wrap is stripped). For the hoisting operands prettier instead relocates the comment out of the operand's parens (`!(/* c */ (x = y))`, `!(/* c */ (cond ? b : c))`) ŌĆö a divergence, see [assignment_operand_leading_comment](../tests/fixtures/typescript/expressions/unary/assignment_operand_leading_comment_prettier_divergence/) and [paren_operand_leading_comment](../tests/fixtures/typescript/expressions/unary/paren_operand_leading_comment_prettier_divergence/); a **type-assertion** operand instead keeps the single pair in *both* formatters (a match ŌĆö [type_assertion_operand_leading_comment](../tests/fixtures/typescript/expressions/unary/type_assertion_operand_leading_comment/)). A *trailing* comment on such an operand does keep both pairs (`!((x = y) /* c */)`, [operand_paren_comment](../tests/fixtures/typescript/expressions/unary/operand_paren_comment/)) ŌĆö the comment-holder path fires on `has_trailing_comments`, so only the leading-comment case collapses to one pair.

**Svelte template / directive positions are not covered by the JS-vs-TS split.** The "JS preserves" rule above holds only inside `<script>` (prettier's babel/oxc path). In a Svelte **template** ŌĆö attribute values (`title={ŌĆ”}`), block tests (`{#if ŌĆ”}`), and mustache tags (`{ŌĆ”}`, `{@html ŌĆ”}`) ŌĆö prettier-plugin-svelte routes the expression through a path that **strips** the cast even in a plain (JS) component. tsv preserves uniformly ŌåÆ divergence, pinned by [jsdoc_cast_template](../tests/fixtures/svelte/syntax/comments/jsdoc_cast_template_svelte_prettier_divergence/) (also a `_svelte_divergence` ŌĆö the cast comment's `leadingComments` attachment differs from Svelte; see [conformance_svelte.md](./conformance_svelte.md) ┬¦Comment Attachment Differences). The `{@const}` case is a prettier **bug**: `{@const y = /** @type {T} */ (z)}` makes prettier-plugin-svelte emit invalid output `(z}` (it drops the `)`) and then throw on its own output; tsv preserves it correctly, so it is documented here but cannot be pinned as an `output_prettier.*` oracle.

#### Comment normalization (stable quirks)

Prettier has multiple stable forms for comment positioning. tsv normalizes to a single canonical form.

- Member access after-call comment ŌĆö [trailing_member_after_call_comment](../tests/fixtures/typescript/expressions/calls/chained/trailing_member_after_call_comment_prettier_divergence/)
- Type-alias union value before-`;` line comment (2-pass: spurious union break, then collapse) ŌĆö [type_alias_value_trailing_comment](../tests/fixtures/typescript/types/comments/type_alias_value_trailing_comment_prettier_divergence/)
- Value-position parenthesized-assignment trailing line comment ŌåÆ Floated out past `;` (`const y = (a = b); // c`); tsv keeps it inside the parens, consistent with the sequence/block cases (prettier itself keeps the *block* case inside) ŌĆö [init_assignment_paren_line_comment](../tests/fixtures/typescript/statements/variable/init_assignment_paren_line_comment_prettier_divergence/)
- Block comment mid-chain ŌĆö [block_comment_chain](../tests/fixtures/typescript/expressions/calls/chained/block_comment_chain_prettier_divergence/)
- Property signature leading line comment ŌĆö [annotation_simple](../tests/fixtures/typescript/types/comments/annotation_simple_prettier_divergence/)
- Property signature leading block ŌĆö [annotation_leading_block](../tests/fixtures/typescript/types/comments/annotation_leading_block_prettier_divergence/)
- Param/destructuring default block, own-line value ŌåÆ tsv collapses to after `=` (`a = /* c */ b`), prettier to before `=` (`a /* c */ = b`) ŌĆö [param_default_own_line_block_comment](../tests/fixtures/typescript/declarations/function/param_default_own_line_block_comment_prettier_divergence/)

**Computed access comment**: Prettier requires 2 passes to stabilize line comments before computed access (`[0]`). The intermediate form places the comment inside brackets (`[// comment\n0]`), which then normalizes to end-of-line (`[0]; // comment`). tsv reaches the stable form in one pass.

**Block comment mid-chain**: When nested grouping parens with block comments are stripped on a member chain (e.g., `/* outer */ (/* inner */ (a).b).c(fn)`), Prettier repositions the inner comment mid-chain (`a /* inner */.b`) and breaks the chain. Prettier requires 2 passes to stabilize the spacing: pass 1 produces `a/* inner */ .b`, pass 2 produces `a /* inner */.b`. Both passes break the chain at the same point. Not JSDoc-specific ŌĆö any block comment before stripped grouping parens triggers this. tsv normalizes directly to the stable form in one pass.

**Property signature leading block**: A block comment between `:` and the type in a property signature has two intentional stable positions (`a: /* block */ X;` after `:`, and `a /* block */: X;` before `:`); both formatters preserve each when given as input. The divergence is in normalizing **unstable** layouts ŌĆö when the user breaks the line around the comment (`a: /* block */\n X;` or `a:\n /* block */\n X;`), tsv compacts to the inline form after `:`, while prettier eventually relocates the block before the `:` (sometimes via a multi-pass convergence). Neither choice is information-destructive; this is purely about which canonical target to favor for ambiguous inputs.

**Param/destructuring default block, own-line value**: The exact analog of the property-signature case at a parameter (or destructuring binding) `=` default. A block comment in the `=`ŌåÆvalue gap has two intentional inline positions ŌĆö `a = /* block */ b` after `=` (on the value) and `a /* block */ = b` before `=` (on the name) ŌĆö and both formatters preserve each when given as input. When the value is authored on its own line (`a = /* block */\n b`), tsv collapses to the after-`=` form while prettier collapses to the before-`=` form. Block comments inline losslessly, so neither wraps; the only difference is which side of `=` the comment lands on for the ambiguous own-line input.

**Property signature leading line comment**: For a line comment between `:` and an inline-renderable type in a property signature (`{ prop: // c\n X }` ŌĆö covers identifiers, optional `?:`, readonly, computed keys, generics like `Array<X>`, tuples, function types, `typeof`, etc.), Prettier moves the comment past the implicit `;` to end-of-line (`prop: X; // c`); tsv keeps the comment after `:` and drops the type to a continuation line indented one level (`prop: // c\n\t X;`, the [Uniform Forced-Continuation Indent](#uniform-forced-continuation-indent)). Both forms are stable under their own formatter. A multi-member **union** in the same position **diverges** ŌĆö prettier drops the comment onto its own line after `:` while tsv keeps it trailing the `:` + indents the union (the [annotation](../tests/fixtures/typescript/types/comments/annotation_prettier_divergence/) fixture); a multi-member **intersection** likewise **diverges** (prettier keeps it flush, tsv indents ŌĆö see [annotation_continuation_indent](../tests/fixtures/typescript/types/comments/annotation_continuation_indent_prettier_divergence/)). Notably, prettier's end-of-line motion is information-destructive when more than one comment touches the property: leading line + trailing line collapses to `f: X; // leading // trailing` (second `//` becomes text inside the first comment); two leading lines merge **and reverse** order (`g: // c1\n // c2\n X;` ŌåÆ `g: // c2 // c1\n X;`); leading line + trailing block reorders to `h: X; /* trailing */ // leading`. tsv preserves each comment at its authored position as a separate comment node. The end-of-line **relocation** is property-signature-only ŌĆö prettier keeps variable declarations (`const e: // c\n X = ...`) and class properties (`class C { prop: // c\n X }`) in place ŌĆö but tsv's continuation **indent** is universal across all these contexts, so those keep-in-place cases become an indent-only divergence too (the same [annotation_continuation_indent](../tests/fixtures/typescript/types/comments/annotation_continuation_indent_prettier_divergence/) fixture).

### Format-ignore directive

A comment can suppress formatting of the construct that follows it. tsv honors its own tool-neutral `format-ignore` family ŌĆö `<!-- format-ignore -->`, `// format-ignore`, `/* format-ignore */`, and the range markers `format-ignore-start` / `format-ignore-end` ŌĆö **in addition to** prettier's `prettier-ignore` family, which tsv keeps for drop-in compatibility (corpus files use it). Recognition is centralized in `tsv_lang::is_format_ignore_directive` and the two range predicates, shared across the TypeScript, CSS, and Svelte printers.

The `prettier-ignore` family matches prettier exactly (both emit the construct raw), so it needs no divergence fixture of its own. For `<script lang="ts">` the plugin formats the embedded script through
prettier's real **`typescript`** parser (`embed.ts` ŌåÆ
`textToDoc(content, {parser: 'typescript'})`), so a construct that path throws
on ŌĆö e.g. `@(a?.b)()` (a `TypeError` in prettier's needs-parens printer) or
`x?.#a` (`An optional chain cannot contain private identifiers.`) ŌĆö **does**
trigger the verbatim fallback in `.svelte`, exactly as on a pure-`.ts` run. (A
`<script>` with **no** explicit `lang` is formatted through `babel-ts` instead,
whose stricter TC39 decorator grammar rejects forms like `@(f()).g` that tsc
and the `typescript` parser accept.) **Both tsv pipelines disarm this with
`PRETTIER_DEBUG=1`** (the tsv_debug sidecar sets it on the Deno spawn; the
`corpus:compare:format:run` task sets it in its env), which makes the plugin
and prettier-core rethrow ŌĆö so `compare`, fixture validation,
`fixtures_update`, and corpus runs all report a hard prettier error (with a
code frame) instead of fake-stable output. The caveat applies when probing
prettier **outside** these pipelines (a bare `prettier` invocation, editor
integrations, upstream issue repros): there the fallback silently "preserves"
the whole script. Confirm by re-running the suspect construct in a single-form
file or as pure `.ts`, where no fallback exists so it fails visibly. (Also see
[fixture_overview.md ┬¦Common Pitfalls](./fixture_overview.md#common-pitfalls) ŌĆö
the fallback can fake a "prettier-stable" fixture input.)
