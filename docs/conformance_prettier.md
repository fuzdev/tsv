# Prettier Conformance

Prettier was tsv's initial guide, and the formatter still tracks it for the common case â€” but tsv has its own identity and makes **intentional, cataloged choices** to diverge where they're more defensible. This document catalogs those divergences along with bugs that tsv does not replicate.

## Terminology

**Matched**: tsv produces identical output to Prettier â€” the goal and the common case (measure current rates with `deno task corpus:compare:format --all --summary`).

**Unmatched**: tsv produces different output. The suffix `_prettier_divergence` marks these fixtures. This document explains WHY for each case.

## Reasons tsv Differs

Each divergence is tagged with one or more `â—†reason` tokens â€” single greppable
units (search `â—†` for every tag, `â—†prettier_bug` for one category):

- `â—†spec_violation` â€” Prettier emits spec-violating CSS/HTML/JS. tsv follows the spec
- `â—†spec_precedence` â€” Prettier's output is valid but tsv emits the spec's canonical serialized form
- `â—†stable_quirk` â€” Prettier preserves multiple forms without normalizing. tsv normalizes consistently
- `â—†prettier_bug` â€” Prettier is non-idempotent, emits invalid output, or changes meaning (e.g. strips required parens). tsv produces stable, valid, meaning-preserving output
- `â—†parser_compat` â€” Prettier's output breaks Svelte's parser. tsv produces Svelte-compatible output
- `â—†print_width` â€” Prettier allows lines to exceed printWidth. tsv breaks to stay within the limit
- `â—†tabs_only_indent` â€” Prettier mixes tabs and spaces under --use-tabs. tsv uses whole tabs only
- `â—†bom_stripping` â€” Prettier preserves byte-order marks. tsv strips them
- `â—†comment_preservation` â€” Prettier moves comments to a different syntactic position. tsv preserves comment position
- `â—†content_preservation` â€” Prettier silently drops user comments. tsv preserves all comments
- `â—†design_choice` â€” Other deliberate behavior differences, with rationale in the fixture

> Most `â—†comment_preservation` and `â—†content_preservation` divergences live in the prose-form [TypeScript: Comments](#typescript-comments) and [CSS: Comments](#css-comments) catalogs, not the tag-prefixed catalog lists â€” they're the largest divergence category but don't reduce to a single bullet.

## Decision Framework

**When to match Prettier:**

- Cosmetic choices (spacing preferences, quote styles)
- Output that's valid and reasonable
- Unclear which approach is "better"

**When to differ:** any reason in [Reasons tsv Differs](#reasons-tsv-differs) above. The three cross-cutting principles â€” comment position, print width, and tabs-only indentation â€” are detailed below.

### Comment Position Philosophy

**A formatter should not move a comment to a different syntactic position â€” unless
the move is lossless and the position carries no authorship signal.** Comment
placement is usually a deliberate authoring choice â€” it communicates what the comment
refers to â€” so preserving it is tsv's default and its single largest category of
divergence from Prettier (see [TypeScript: Comments](#typescript-comments)).

Prettier's comment handling is its weakest area. It routinely moves comments from
between syntactic boundaries into adjacent blocks, parens, or other positions, changing
the apparent association â€” and frequently **losing information** (two comments merging
onto one end-of-line, the second `//` becoming text; or reordering them). tsv treats
comment position as semantic and preserves it wherever that distinction is real.

**Principles:**

1. **Comments between an operator and its operand stay there.** If the user wrote
   `? foo : // about bar`, the comment stays after `:`. Prettier moves it to trailing
   on `foo`, changing its association from the false branch to the true branch.
2. **Trailing comments stay trailing.** `foo // comment` keeps the comment on `foo`.
3. **Same-line block comments stay same-line.** `extends T /* c */ ?` keeps the
   comment after `T`, not moved after `?`.
4. **Both positions are valid when dual-stable.** When the user's chosen position is
   idempotent, preserve it. Don't collapse to one canonical form â€” that destroys the
   distinction between "comment about X" and "comment about Y".
5. **The deciding test is information loss, not position purity.** Preserve a comment's
   position when relocating it would lose information â€” the canonical case is Prettier's
   end-of-line relocation *merging* two comments onto one line (the second `//` becomes
   text) or reordering them; tsv keeps them distinct (the nameâ†’`=`/`:`/`?` binding
   divergences). But where relocating is **lossless** *and* the position carries no
   signal â€” a same-line line comment past a *pure separator*, e.g. a list element's
   comma (`A // câŹŽ, B` â†’ `A, // c`; the comma is structure and the comment trails the
   element either way) â€” tsv trails like Prettier rather than manufacturing a divergence
   for a meaningless distinction.

**When reviewing comment-related fixes:** Default to preserving position. Match
Prettier's repositioning only when the move is lossless *and* the position carries no
authorship signal (a pure-separator trail), or when the original position is clearly
wrong (e.g., comment inside a token boundary). Otherwise â€” and whenever relocating
would merge, reorder, or drop a comment â€” preserve and create a `_prettier_divergence`
fixture.

### Uniform Forced-Continuation Indent

A direct corollary of comment-position preservation, and tsv's most cross-cutting
comment-layout rule. When a **line** comment forces part of a construct onto a new
line â€” a `//` runs to end-of-line, so whatever the author wrote after it cannot stay
on that line â€” tsv keeps the comment where it was written and drops the following
token to a continuation line **indented one level**. The continuation then reads as
part of its construct, not as a sibling statement or member.

One rule, applied at every site where a line comment splits a construct's head from
its tail:

- **Declaration and module headers** â€” keywordâ†’name, keywordâ†’`{`, bindingâ†’`from`,

  comment between a key/binding name (or its `?`/`!` marker) and the `:`

The indent is tsv's own layout choice; prettier handles each site differently â€” it
relocates the comment (into braces/parens, after `from`/`as`/`;`), floats it past
`;`, keeps the continuation **flush**, or drops the comment onto its own line. So

regular fixture; the multi-member union after `:` is a divergence (prettier drops the comment

construct the comment split.

### Print Width Philosophy

**Prettier treats `printWidth` as a soft target.** Lines may exceed it in various edge cases (fill algorithm boundaries, block expressions, template literals, certain constructs that "don't look good" when wrapped).

**tsv treats `printWidth` as a hard limit.** If content exceeds 100 characters, tsv breaks it when possible. This is a deliberate design choice affecting many divergences in this document:

- Block expression conditions (`{#if}`, `{#each}`, etc.) with logical operators
- Template literal interpolations
- Fill algorithm edge cases (101-char boundaries)
- Single-specifier imports
- Various "short expression" tolerances

The benefit: predictable output that respects the configured line length. The tradeoff: some constructs may break where Prettier would keep them inline.

### Tabs-Only Indentation Philosophy

**tsv renders all indentation as whole tabs and never mixes tabs with alignment spaces.**
When breaking a union type, Prettier wraps each member's type doc in
`align(2, â€¦)` (`union-type.js`) to offset the 2-char `|` prefix. With
`--use-tabs`, Prettier's indentation algorithm renders that 2-column offset as
a **sub-tab alignment**: content that is further indented rounds the offset up
to a whole tab, but a closing delimiter sitting at the offset column is emitted
as `tabs + 2 spaces`. The result mixes tabs and spaces on a single indentation
level.

tsv rounds the 2-column offset up to one tab everywhere. At
`tabWidth = 2` the two are the same visual width; only the representation
differs (`âź¨n+1 tabsâź©}` vs `âź¨n tabsâź©Â·Â·}`). This keeps indentation
tab-width-agnostic: a reader viewing tabs at any width sees consistent
structure, whereas Prettier's 2-space offset assumes the prefix is exactly 2
columns wide. Cataloged in [Tabs-Only Alignment](#tabs-only-alignment).

**Spec violations**: CSS Syntax 3 Â§4.3.4 specifies that an identifier immediately followed by `(` tokenizes as a `<function-token>`, not as an `<ident-token>` plus `(`. Media Queries 4 Â§3 explicitly notes: "Whitespace is required between a 'not', 'and', or 'or' keyword and the following '(' character, because without it that would instead parse as a `<function-token>`." Container Queries (CSS Conditional 5) use the same grammar pattern. Prettier normalizes this for `@supports` but not `@media` or `@container`.

**SCSS directive numbers**: SCSS/Sass directives (`@include`, `@mixin`, `@if`, `@for`, `@each`, `@while`, `@function`, `@return`, `@debug`) are not standard CSS. Per CSS Syntax 3 Â§5.4.2 an unknown at-rule's prelude is consumed as an opaque list of component values with no defined grammar, so tsv preserves it verbatim (e.g. `@include foo(.5s)` stays `.5s`, `@include baz(1.50)` stays `1.50`). **Media comma-list wrapping (not a divergence)**: a comma-separated `@media` query list (Media Queries 4 Â§"media query list") that exceeds print width is broken at every top-level comma â€” one query per line, one indent level â€” matching Prettier exactly. **CSS keyword case â€” the canonicalize/preserve line.** CSS keywords are ASCII case-insensitive (CSS Syntax 3 Â§"tokenizing"). tsv adopts prettier's canonicalization set wholesale â€” it **lowercases** property names, at-rule names, units, pseudo-class/element names, media-feature names, keyframe `from`/`to`, hex, and `!important`, and **preserves** values (`color: RED`), custom properties (`--MyVar`), type/element selectors (`DIV`, SVG `linearGradient`), function names (`URL(`/`RGB(`), media types (`SCREEN`), and the `not`/`only` media qualifiers â€” matching prettier on both sides. tsv parses every case-insensitive keyword (so `@supports (a: b) AND (c: d)`, `@media screen AND (â€¦)`, `:NTH-CHILD(2n+1)` all parse â€” Class-1 correctness) but only rewrites the spelling where prettier does. ### Svelte: Inline content block-style

Design choice. tsv lays out an inline element's **wrapping content** block-style â€” both tags stay intact and the content moves to its own indented line(s), collapsing to `<tag>content</tag>` when it fits, exactly like a block element. Prettier instead **dangles** the tag delimiters (`<tagâŹŽ\t>content</tagâŹŽ>`, pre-breaking the opening tag and dangling the closing `>`). This is uniform across text, expression, and element/component children, with and without attributes (the `>` is **attr-keyed** â€” it hugs the last attribute when attributes fit and dedents to its own line when they wrap), and including table cells (which are inline-classified). Content that fits stays inline; only overflow breaks to block-style.

Content-boundary whitespace is **render-free under Svelte 5** â€” start/end-of-content whitespace is trimmed at compile (`<p>foo<span> - bar</span></p>` renders `foo- bar`), so the injected block-style boundaries are render-equivalent (confirmed at corpus scale by `ast_diff --render`). **Constrained infer extends-operand parens**: An `infer X extends C` only ever appears in a conditional type's extends-type, so a trailing token always follows the constraint. ### TypeScript: Template Literals

tsv formats template interpolation `${...}` using two strategies based on expression type:

- **Qualifying types** (Identifier, MemberExpression, ConditionalExpression, BinaryExpression, SequenceExpression, TSAsExpression, TSSatisfiesExpression, etc.): softline wrapping at `${`/`}` boundaries â€” the group breaks when the line exceeds print width. This provides width enforcement for expressions that have no internal break points.
- **Non-qualifying types** (CallExpression, chains, ArrowFunction, etc.): no softlines at `${`/`}` â€” expression breaks internally while `${`/`}` stays hugged. Matches Prettier's approach where these types keep their doc structure.

Prettier uses atomic text (no doc structure) when the expression has no structural newlines, which means internal breaks can't happen. tsv preserves doc structure, so non-qualifying expressions can still break internally when they exceed width. This divergence appears when non-qualifying expressions exceed width and would need internal breaks.

- Multiline indent â€” [interpolation_multiline_indent_long](../tests/fixtures/typescript/expressions/literals/template/interpolation_multiline_indent_long_prettier_divergence/)
- Nested template â€” [interpolation_nested_template](../tests/fixtures/typescript/expressions/literals/template/interpolation_nested_template_prettier_divergence/)
- Template literal type â€” [template_literal_type_long](../tests/fixtures/typescript/types/template_literal_type_long_prettier_divergence/)
- Template literal type (multibyte width) â€” [template_literal_type_multibyte_long](../tests/fixtures/typescript/types/template_literal_type_multibyte_long_prettier_divergence/)

**Expression atomization**: Prettier pre-renders each template expression at `printWidth: Infinity` (`template-literal.js:212-226`). If the rendered result is single-line (no newlines), prettier replaces the expression doc with an atomic string â€” making it impossible to break, even when the line exceeds print width. Only expressions that naturally produce multi-line output get softline wrapping. tsv always preserves doc structure, so qualifying expressions can break at `${`/`}` boundaries when the line exceeds width. This is the primary source of template literal divergences: simple expressions like `${prop}` or `${obj.field}` stay inline in prettier (atomic) but may break in tsv (softline group).

**Multiline indent**: For code-generation templates with indented content, tsv applies Prettier's `addAlignmentToDoc` for indent calculation (using ceiling division to match prettier's useTabs rounding). Non-qualifying types (chains, calls) hug at `${}` with internal breaks, while qualifying types break at `${`/`}` when exceeding width.

**Nested template**: When a template expression contains an array literal wrapping a long inner template, tsv breaks the array to respect print width while Prettier keeps it inline.

**How tsv treats keywordâ†’value block comments** (the gap between a keyword/operator and its **single operand** â€” `as`/`satisfies`, heritage/conditional `extends`, `keyof`/`typeof`/`readonly`, `infer`, a type parameter's `extends`/`=`, predicate `is`, and indexed access `[`â†’index). These are **single-slot gaps**: one keyword/bracket, one operand, no delimited list. The governing principle across the whole comment story is *per construct, emit the least-divergent output that is lossless, idempotent, **and** position-preserving*; at a single-slot gap that lands on **collapse** â€” a block comment collapses to the inline form (`kw /* c */ operand`) whenever it *can* inline losslessly: a **single-line** block in any position (glued, trailing the keyword, or own-line), and a **glued** multiline block, one whose operand already shares the comment's closing line (`kw /* â€¦âŹŽâ€¦ */ operand`, the way Prettier keeps it â€” [keyword_value_glued_multiline_block_comment](../tests/fixtures/typescript/types/keyword_value_glued_multiline_block_comment/)). The own-line placement of such a block carries no signal the inline form drops, so tsv keeps the comment after the keyword and collapses. Only a **line** comment (a `//` would swallow the operand) or an **own-line** multiline block (`kwâŹŽ/* â€¦âŹŽâ€¦ */âŹŽoperand` â€” inlining it would reflow the author's break) still hangs the operand on its own line, and there tsv keeps the comment where the author wrote it rather than pulling it onto the keyword line ([type_operator_keyword_own_line_multiline_block_comment](../tests/fixtures/typescript/types/type_operator_keyword_own_line_multiline_block_comment_prettier_divergence/) for the operator family). The gate is `comments_force_own_line_between` (hang iff a line comment, or a multiline block with a newline after it), shared with the type-alias `=` layout â€” whose `=` stays on its line for a glued multiline value too, rather than breaking on the comment's interior newline.

Collapse is not a universal law over "single-line block in a bracket" â€” it's the outcome of that principle *at a single-slot gap*. A delimited `<â€¦>` list is a different construct and takes a different outcome (see the list contrast below). Within the gaps, the divergence lives entirely in the **own-line authoring** (`kwâŹŽ/* c */âŹŽoperand`), which the two formatters normalize to different fixed points â€” each fixture's `unformatted_ours_own_line` variant pins tsv's collapse. The two flavors of Prettier fixed point:

- **In-place-collapse gaps.** Prefix type operators `keyof`/`typeof`/`readonly` ([type_operator_keyword_own_line_block_comment](../tests/fixtures/typescript/types/type_operator_keyword_own_line_block_comment_prettier_divergence/)) and `infer` ([keyword_own_line_block_comment](../tests/fixtures/typescript/types/infer/keyword_own_line_block_comment_prettier_divergence/), line form [keyword_line_comment](../tests/fixtures/typescript/types/infer/keyword_line_comment_prettier_divergence/)): Prettier reaches the *same* inline form, but is **non-idempotent** â€” its first pass pulls the comment onto the keyword line yet leaves the operand on the next line, collapsing fully only on a second pass.
- **Relocation gaps.** `as`/`satisfies` ([as_satisfies_value_own_line_block_comment](../tests/fixtures/typescript/expressions/as_satisfies_value_own_line_block_comment_prettier_divergence/)), type predicate `is` ([predicate_is_own_line_block_comment](../tests/fixtures/typescript/types/predicate_is_own_line_block_comment_prettier_divergence/)), a type parameter's `extends`/`=` ([constraint_default_own_line_block_comment](../tests/fixtures/typescript/types/type_params/constraint_default_own_line_block_comment_prettier_divergence/)), a conditional type's `extends` ([extends_own_line_block_comment](../tests/fixtures/typescript/types/conditional/extends_own_line_block_comment_prettier_divergence/)), and indexed access `[`â†’index ([indexed_access_own_line_block_comment](../tests/fixtures/typescript/types/indexed_access_own_line_block_comment_prettier_divergence/)): Prettier collapses *and relocates* the comment across the keyword/bracket (`x /* c */ as A`, `A /* c */[K]`), and reaches that relocation **non-idempotently** in every case â€” its first pass leaves the comment after the keyword and hangs the operand (`x as /* c */âŹŽA`, `A[/* c */âŹŽK]`), only its second pass moves the comment across (`x /* c */ as A`, `A /* c */[K]`). tsv keeps it after the keyword and collapses in one pass â€” the pure [comment-position](#comment-position-philosophy) divergence. Each fixture pins Prettier's two-pass chain with a `prettier_intermediate_to_variant_own_line` (its unstable first pass) plus a `variant_own_line` (its relocated fixed point, dual-stable in both formatters).
- **Heritage `extends`/`implements`** ([heritage_keyword_own_line_block_comment](../tests/fixtures/typescript/class/heritage_keyword_own_line_block_comment_prettier_divergence/)): also a relocation gap, but Prettier does **not** collapse it. From the own-line authoring it settles â€” in one pass, stably â€” on the comment on its **own line before** the keyword (`class PâŹŽ/* c */âŹŽextends Q`), never inlining it. tsv collapses to `class P extends /* c */ Q` (after the keyword, inline). So both move the comment relative to the author's break, but Prettier keeps the break while tsv removes it. Prettier's stable form is *not* dual-stable â€” tsv re-collapses it to a *third* form (`class P /* c */ extends Q`) â€” so it is pinned as a `divergent_variant_*` form rather than a `variant_*` (see the note below).

The own-line authoring of a *single-line* block is the only place these gaps diverge; the inline canonical (`kw /* c */ operand`) is dual-stable in both formatters. Only a **line** comment (a `//` can't be inlined â€” it would swallow the operand) or an **own-line multiline** block comment still hangs the operand on its own line; the collapse applies to single-line blocks (any authoring) and glued multiline blocks. The companion **line**-comment divergences (where the hang is forced) are cataloged alongside each gap above.

**List contrast â€” why `<â€¦>` does not collapse.** A delimited type-argument list `<â€¦>` is *not* a single-slot gap, so it takes the other outcome of the same principle: Prettier has a clean, idempotent list-break rule (an own-line comment on an argument breaks the whole `<â€¦>`), and matching it is the least-divergent lossless choice â€” so tsv matches instead of collapsing. A single-argument own-line *block* comment therefore **expands** (`Foo<âŹŽ/* c */âŹŽTâŹŽ>`, dual-stable in both formatters â€” [single_arg_own_line_block_comment](../tests/fixtures/typescript/types/type_args/single_arg_own_line_block_comment/)) rather than collapsing to `Foo</* c */ T>`; the single-argument case is just the N=1 form of the list and follows the list, not the gap. (A single-argument leading *line* comment hugs `<`/`>` â€” a separate divergence, cataloged under the [open-delimiter trailing](#comment-relocation) entry.) The surface split â€” index `[K]` collapses, `<T>` expands â€” is not an inconsistency: it is the same "diverge only when Prettier is worse" rule landing on *collapse* where Prettier relocates non-idempotently (the index slot) and on *match* where Prettier is already fine (the list).

Two carve-outs keep the operand break instead of collapsing: **`export default`** still hangs (it's the one construct where Prettier *keeps* the operand break, so hanging â€” both broken, tsv indented â€” is the closer indent-only divergence) and **binary/logical operands** (the expression side, where tsv's own-line layout already matches Prettier). The **conditional `?`/`:` branches** collapse the single-line and glued forms and match Prettier there (no divergence fixture). Two glued-multiline layout gaps remain, unrelated to the block-comment gate and unchanged here (tracked follow-ups): the conditional `?`/`:` *branch arm* still over-hangs a **glued** multiline block (`? /* â€¦âŹŽâ€¦ */âŹŽA` where Prettier keeps `? /* â€¦âŹŽâ€¦ */ A`), and a class/interface **heritage** clause keeps a glued multiline inline (`class P extends /* â€¦âŹŽâ€¦ */ Q`) where Prettier breaks before the keyword. **Pinning Prettier's fixed point (fixture mechanics).** Because each inline `input` is dual-stable, these fixtures have no `output_prettier` â€” the divergence lives only in the `unformatted_ours_own_line` variant (tsv normalizes it to `input`; Prettier does not). In a **default** (`import /* c */ Foo from 'a'`) or **namespace** (`import /* c */ * as ns from 'a'`) import, tsv keeps each header comment where the user wrote it; Prettier keeps a comment already adjacent to the binding but relocates a comment between `import` and `type` to the binding side of `type` (`import type /* c */ Foo`). In an **export-all** (`export /* c */ * from 'a'`, including `export /* c */ type * from 'a'`) Prettier relocates _every_ header comment â€” around `export`, `type`, and `*` â€” to after `from`, before the source; tsv preserves them in place. A comment between `*` and `as` in a namespace binding (`import * /* c */ as ns`, `export * /* c */ as ns from 'a'`) is likewise preserved in place â€” Prettier relocates it to after `as`, before the binding. Both positions are dual-stable in our formatter. Per the uniform module-header rule below, a line comment in **every** one of these gaps indents the continuation one level â€” the keywordâ†’default/namespace binding, `type`â†’default/namespace binding, `type`â†’namespace-`*`, export-all `export`/`type`â†’`*`, `*`â†’`from`, `*`â†’`as`, and `as`â†’binding gaps alike. Where Prettier relocates (export-all, `*`â†’`as`) tsv is free; where Prettier keeps the comment in place and flat (keyword/`type`â†’binding, `type`â†’namespace-`*`, `as`â†’binding) tsv's indent is a deliberate indent-only divergence.

**Import/export binding-or-specifiers to `from`**: A comment in the gap between an import's binding/specifiers (or a re-export's specifiers) and the `from` keyword is preserved where the user placed it; Prettier's relocation depends on the binding shape. For a **default** or **namespace** binding (no braces) a same-line block comment stays in place in both formatters (`import Foo /* c */ from 'a'` â€” dual-stable), but Prettier floats a line comment past the `;` to a statement-trailing position (`import Foo from 'a'; // c`, Prettier's statement-trailing float) while tsv keeps it before `from`, indenting the `from â€¦` continuation one level onto its own line. For **named specifiers** (`import { a } /* c */ from 'a'`, `export { a } /* c */ from 'a'`) Prettier relocates the comment _into_ the braces as the last specifier's trailing comment â€” a block comment inline (`{ a /* c */ }`), a line comment expanding the braces multiline (`{\n\ta // c\n}`) â€” while tsv keeps it after `}` with the braces inline (indenting `from â€¦` when a line comment forces the break). Both positions are dual-stable in our formatter. (Otherwise the comments are dropped entirely â€” content loss.)

**Import attributes header comments**: A comment in an import's attributes header â€” between the source and the `with` keyword (`import x from 'a' /* c */ with {â€¦}`), or between `with` and the attributes `{` (`import x from 'a' with /* c */ {â€¦}`) â€” is preserved where the user placed it. Prettier keeps a sourceâ†’`with` block comment in place (dual-stable), relocates a `with`â†’`{` block comment to before `with` (`import x from 'a' /* c */ with {â€¦}`), and floats a `with`â†’`{` line comment past the `;` (`import x from 'a' with {â€¦}; // c`, Prettier's statement-trailing float); for a sourceâ†’`with` _line_ comment, Prettier collapses `with {â€¦}` back onto the source line and floats the line comment past the `;` (`import b from './b' with {â€¦}; // c`), while tsv keeps it before `with` â€” see [with_keyword_comment_line](../tests/fixtures/typescript/modules/imports/with_keyword_comment_line_prettier_divergence/). tsv keeps each comment where the user wrote it; when a line comment forces the `with`/`{â€¦}` onto a new line, tsv indents that continuation one level. The attributes `}`â†’`;` block comment trails after the `;` in both formatters (the **Import source to `;`** entry below). (Otherwise the header comments are dropped entirely â€” content loss.)

**Keyword â†’ specifier brace comments**: A comment between the `import`/`export` keyword and the named-specifier `{` (`import /* c */ { a }`, `export /* c */ { a }`) is preserved before the brace; Prettier relocates it _into_ the braces as the first specifier's leading comment â€” a block comment inline (`import { /* c */ a }`), a line comment expanding the braces multiline. tsv keeps it where the user wrote it; a line comment forces `{` onto the next line, indenting the `{â€¦}` continuation one level (the uniform module-header rule below). (The `import type â€¦ { a }` typeâ†’`{` gap and the empty-braces `import /* c */ {}` gap are preserved the same way; this is the non-type named-specifier case.) **Declaration- and module-header line-comment continuation indent**: A _line_ comment in a declaration- or module-header gap forces the following token onto a new line, and tsv indents that continuation one level â€” a statement spanning lines reads as a continuation, not a second statement.

For **module headers** tsv applies this **uniformly to every gap, with no exceptions**: keywordâ†’`type`, keyword/`type`â†’default-binding, keyword/`type`â†’namespace-`*`, keyword/`type`â†’`{`, keyword/`type`â†’empty-`{}` (re-export _and_ no-`from`), bare keywordâ†’source, export-all `export`/`type`â†’`*`, `*`â†’`from`, `*`â†’`as`, `as`â†’binding, binding/specifiersâ†’`from`, `from`â†’source, the `with`/attributes-`{` gaps, and `export`/`export default`â†’declaration. Prettier's own handling varies per gap â€” it relocates the comment (into the braces, after `from`/`as`/`;`), floats it past `;`, or keeps it in place and flat â€” but tsv **always** indents. So where Prettier relocates, tsv's in-place indent is just one of two unrelated layouts; and where Prettier keeps the comment in place and flat (`as`â†’binding, keyword/`type`â†’default/namespace binding, `type`â†’namespace-`*`, bare/empty/export-all `from`â†’source, no-`from` empty `{}`, `export`â†’declaration) tsv's indent is a deliberate indent-only divergence, chosen so every continuation reads alike.

For **declaration headers** the same rule covers the keywordâ†’name gap of `function` (incl. `async function`, `function*`, and `declare function`), `class` (incl. `abstract class`), and `enum` (incl. `const enum`), plus the keywordâ†’declarator gap of `const`/`let`/`var`. Prettier keeps the comment in place and flat for the `function`/`class`/`enum` headers, so tsv's indent is an indent-only divergence there; for `const`/`let`/`var` Prettier **agrees** (it also indents the declarator), so there is no divergence â€” a regular fixture.

The deliberate exclusions are constructs where the following token isn't part of the same declaration: **ASI statement keywords** (`return`, `throw`, `break`, `continue`, `yield`) â€” a newline after them triggers automatic semicolon insertion, so `return // câŹŽexpr` is two statements, not a continuation; the **contextual declaration keywords** (`type`/`interface`/`namespace`/`module`/`declare`), whose line forms never form a single declaration â€” the keyword requires its following token on the **same line** (tsc's `isDeclaration`: `nextTokenIsIdentifierOnSameLine`, and the ASI modifier rule for `declare`), so a line break demotes the keyword to a plain identifier. `interface`/`namespace`/`module`â†’name then become unparseable (`interface // câŹŽI {}` is rejected); `type`â†’name and `declare`â†’keyword **ASI-split** into two statements (`type;` then `T = â€¦`; `declare;` then `const x = â€¦`). tsv's parser conforms here â€” see [contextual_keywords/declaration_keyword_own_line](../tests/fixtures/typescript/syntax/contextual_keywords/declaration_keyword_own_line/); and **function/class expressions** (`const a = function // câŹŽf() {}`), which are not declaration headers and stay flat in both formatters. Block comments and the no-comment case are byte-identical in every in-scope gap. (The `export`-prefixed forms â€” `export interface // câŹŽI` etc. â€” are out of scope: after `export` the parser is committed to a declaration, where tsc does not apply the same-line gate, so tsv's acceptance there may be spec-correct; acorn-typescript rejects them, but that is a separate acorn-vs-tsc question.)

**Keyword-paren comments**: `if/* c */(a)` â†’ Prettier absorbs comments between keywords and `(` into the condition parens. Applies to `if`, `while`, `for`, `switch`, `catch`, `do...while`. tsv preserves the comment between keyword and paren: `if /* c */ (a)`. For `for await`, comments in either gap are preserved: `for /* a */ await /* b */ (x of y)` â†’ prettier absorbs both into parens. **Condition-absorbed comments**: `switch(x)/* c */{}` â†’ Prettier absorbs comments between `)` and `{` into the condition parens: `switch (x /* c */) {}`. **Type alias head to `=`**: `type A<X>\n// c\n= B | C` â†’ Prettier relocates a line comment between a type alias head (name + optional type parameters) and `=` to after `=` (`type A<X> =\n// c\nB | C`). tsv preserves it before `=`, keeping the comment's association with the declaration head rather than the value. **Type parameter keyword to value (own-line)**: an own-line line comment between a type parameter's `extends`/`=` keyword and its constraint/default value (`U =\n// c\nV`, `T extends\n// c\nA`) is kept on its own line in the indented value block; Prettier pulls the first leading comment up onto the keyword line (`U = // c\n\tV`) and is **non-idempotent** doing so (its first pass leaves the value at the param indent, a second pass adds the extra indent). tsv stays idempotent and preserves the author's own-line placement. **Arrow body stripped parens**: `() => (x /* c */)` â†’ Prettier strips the parens and trails the block comment on the body (`() => x /* c */`); a line comment is floated past `;` (`() =>\n\tx; // c`). **For empty clauses**: Prettier puts comments in syntactically broken positions outside the parentheses.

**Own-line line comment before `;`, blank line after**: a same-line block comment after an import's source / `with {â€¦}` attributes, and a same-line trailing line comment, both trail **after** the `;` in both formatters (`import { a } from './a'; /* c */`, `import { b } from './b'; // 1`) â€” tsv matches prettier here, no divergence. **Try/catch/finally before `{`**: Prettier absorbs line comments between keyword/paren and block into the block body. For catch, the comment gets absorbed into the parameter parens (`catch (\n\te // comment\n)`).

**Between `}` and `else`**: Prettier keeps an **own-line** block comment between the closing brace and `else` where the author wrote it (`}\n/* c */\nelse` â€” a match with tsv, no oracle). With **multiple** inline block comments (`} /* c1 */ /* c2 */ else`), prettier drops every comment after the first onto its own line while tsv keeps them all inline. tsv preserves the comments before the `else` keyword on their own line.

**Trailing member chain**: When line comments appear before trailing member access in chains (`.length` after `.filter()`), Prettier relocates the comment to after the `=` and keeps the chain inline. For a plain member (`.length`) it puts the comment on its own line under `=`; for an **optional**-chain trailing member (`?.length`) it instead trails the comment on the `=` line itself (`const b = // comment`) and de-indents the value. tsv keeps the comment before the trailing member with the chain broken in both cases.

**Block comment in computed `[]`**: When a computed member access with a block comment exceeds print width, Prettier hoists the comment from inside the brackets to before the member chain. Not idempotent.

**Switch case colon comment**: Prettier relocates comments near the colon in switch cases: `case 1: /* c */` â†’ `case 1 /* c */:` (moved before colon); `default /* c */:` â†’ `default: /* c */ break;` (moved to body). tsv preserves comment placement. Three stable forms exist: our input form, prettier's `output_prettier` form, and a third body-only form.

**Class property definite `!`**: `d! /* c */ = 1;` â†’ Prettier moves comment before `!` (`d /* c */! = 1;`). tsv preserves comment after `!`.
**Class property modifier**: Both positions are dual-stable in both formatters. Our canonical form puts the comment after the modifier (`a? /* c */ = 1;`), Prettier's canonical form puts it before (`a /* c */? = 1;`). The user's chosen position is preserved. Same for `!`. Note: when a type annotation follows (`a /* c */?: T`), preserving the comment is a correctness matter, not a divergence â€” it is otherwise dropped entirely (content loss).

**Interface member after `?`**: `a? /* c */ : number;` â†’ Prettier moves before `?` (`a /* c */?: number;`). `b? /* c */(x): void;` â†’ Prettier moves inside parens (`b?(/* c */ x): void;`). tsv preserves both after `?`.
**Type-literal member after `?`**: the type-literal counterpart of the interface case â€” `type T = { a? /* c */ : number }` â†’ Prettier moves before `?` (`a /* c */?: number`), and `m? /* c */(x): void` â†’ inside the parens (`m?(/* c */ x): void`). tsv preserves both after `?`, the same way the interface arm does (the two type-element printers split around `?` consistently). Both positions are dual-stable. A method that has **type parameters** is not a divergence: there Prettier keeps the comment after `?` (`m?/* c */ <T>(â€¦)`), so both formatters agree â€” a regular fixture pins it together with a comment *inside* `<>` (`m<T /* c */>`, also kept in place by both): `types/type_members/method_type_params_comment`.

**Optional `?` to `:` line comment**: a line comment in the gap between an optional `?` and the member's `:` annotation (`a? // c\n: number`) â†’ Prettier relocates it to trail the member `;` (`a?: number; // c`). tsv preserves it after `?`; because a line comment must end its line, the annotation is forced onto the next line. Applies to interface members, type-literal members, and class properties. Preserving the comment is also a content-loss fix: emitting it inline would swallow the `: number` annotation as comment text (non-idempotent).
**Member key to `:` line comment (non-optional)**: the no-marker counterpart of the case above â€” a line comment between a (non-optional) property key and its `:` annotation (`a // c\n: number`) â†’ Prettier relocates it to trail the member `;` (`a: number; // c`). tsv preserves it after the key, the annotation forced onto the next line. Applies to interface members, type-literal members, and class properties. Like the optional case this is also a content-loss fix (emitting the line comment inline would swallow the `: number` annotation â€” non-idempotent); a same-line block comment in the same gap stays inline in both formatters (`a /* c */: number`) and is not a divergence.
**Variable definite `!`**: `let a /* c */!: number;` â†’ Prettier moves comment after `!` (`let a! /* c */ : number;`). tsv preserves before `!`.
**Function param optional `?`**: `function fn(a /* c */?: number) {}` â†’ Prettier moves comment after `?` (`function fn(a? /* c */ : number) {}`). tsv preserves before `?`. Same pattern applies to arrow function params.

**Computed key after `]`**: `[x] /* c */ = 1` â†’ Prettier moves inside brackets (`[x /* c */] = 1`). Applies to object literals, class members, interface members, and destructuring patterns. For interface `set` accessors, prettier moves into params instead (`set [x](/* c */ a)`). tsv preserves between `]` and the next token.

**Heritage last item before `{`**: `class A implements I, J // c {}` â†’ Prettier relocates line comment from after the last heritage item into the class/interface body (`J {\n\t// c\n}`). tsv preserves the comment before `{` with a forced line break (`J // c\n{}`). When more than one comment precedes `{`, each is kept on its own line (`J // c1\n// c2\n{}`) â€” collapsing them onto the heritage line would absorb a following comment into the first line comment's text (`// c1 // c2` reparses as one comment), a content/boundary loss, not just a position change. Affects class `implements`, class `extends`, class expressions, and interface `extends`; the same preservation applies to the interface name/type-paramsâ†’body gap when there is no `extends`. Consistent with tsv's handling of line comments before block bodies across all statement types.

**Type params to `(`**: `<T> /* c */(x: T)` â†’ Prettier moves inside parens as leading comment on first param (`<T>(/* c */ x: T)`). Prettier's behavior is context-dependent: for function declarations/expressions and class methods **with a body**, prettier preserves the comment between `>` and `(` (no divergence). For body-less declarations (overloads, abstract methods, interface/type literal method signatures, call/construct signatures, function/constructor types, declare functions) and arrow functions, prettier moves the comment inside parens. tsv preserves between `>` and `(` in all cases. Both positions are dual-stable.

**Union infix `|` line comment**: `A | // c\n B` (a line comment trailing the infix `|`) â†’ Prettier relocates the comment to trail the previous member (`| A // c\n| B`). tsv keeps it on the separator/`B` side, on its own line so the pipe stays attached (`| A\n// c\n| B`) â€” the comment sits after the `|`, so tsv associates it with `B` rather than `A`. Both forms are dual-stable; the divergence is in how the infix-pipe input normalizes.

**Retained paren union member comment**: a block comment inside a parenthesized union member whose parens are **retained** â€” because the member nests in an outer union or intersection (`a | (b | c /* c */)`, `(a | b /* c */) | c`, `a & (b | c /* c */)`) â†’ Prettier hoists a **trailing** comment out of the parens to after `)` (`a | (b | c) /* c */`, `(a | b) /* c */ | c`, `a & (b | c) /* c */`), but leaves a **leading** comment inside (`a | (/* c */ b | c)` â€” a match with tsv). tsv keeps every such comment inside the parens, associating it with the parenthesized member. (Otherwise the comment is dropped entirely â€” content loss.) When the parens are redundant and stripped â€” a top-level or single-member union â€” both formatters keep the comment in place (no divergence; see `union_intersection_parens_comment`). The parenthesized-_intersection_-in-union member (`a | (b & c /* c */)`) already preserves in place through a separate path. Both positions are dual-stable in our formatter.**Retained paren union member line comment**: the line-comment analog of the above â€” a line comment trailing the last inner member of a retained parenthesized union (`(a | b // c) | c`, `a | (b | c // c) | d`) â†’ Prettier hoists it out to trail the whole member and keeps the inner union inline (`| (a | b) // c`). tsv keeps it inside the parens; because a line comment must end its line, the parenthesized union expands to its broken form (one member per line) with `)` on its own line. (Otherwise the comment is dropped entirely â€” content loss.) Unlike the block-comment case â€” which stays inline because a block comment can â€” the line comment forces the expanded layout. Our expanded form is stable in tsv; Prettier's inline-with-relocated-comment form is its own stable shape.

**Retained paren first-member leading line comment**: a **leading** line comment inside a retained parenthesized union member, when that member is the **first** member of the outer union (`(// c\n A | B) | C`) â†’ Prettier moves the comment out of the parens to lead the member, keeping the inner union inline when it fits (`| // c\n (A | B)`). tsv keeps it inside the parens leading the inner union; because a line comment must end its line, the parenthesized union expands to its broken form with `)` on its own line. (Otherwise the comment is dropped entirely â€” content loss: the inner-leading line comment has no previous member to relocate onto.) This is the leading-comment counterpart of the trailing **Retained paren union member line comment** above, and mirrors its keep-inside behavior. A leading line comment inside a **later** member's parens instead relocates to trail the previous member, where both formatters agree (see the [Tabs-Only Alignment](#tabs-only-alignment) `union_paren_member_long_line_comment` fixture); only the first member, lacking a previous member, keeps the comment inside.

**Retained paren intersection member comment**: the intersection counterpart of the retained-paren-union case â€” a block comment inside a parenthesized **intersection** member whose parens are retained because it nests in an outer union (`(a & b /* c */) | c`, `a | (/* c */ b & c)`, `a | (b & c /* c */)`) â†’ Prettier hoists the comment out of the parens (trailing after `)`, leading before `(`: `(a & b) /* c */ | c`, `a | /* c */ (b & c)`). tsv keeps it inside the parens, associating it with the parenthesized member. (Unlike the union case this never dropped â€” it preserves through the paren-unwrapping path â€” but it is the same comment-position divergence and is pinned for completeness.)
#### JSDoc / paren semantics

- JSDoc type cast parens (standalone TS) â€” [jsdoc_type_cast_ts](../tests/fixtures/typescript/syntax/comments/jsdoc_type_cast_ts_prettier_divergence/)
- JSDoc type cast in Svelte template / directive positions â€” [jsdoc_cast_template](../tests/fixtures/svelte/syntax/comments/jsdoc_cast_template_svelte_prettier_divergence/)

**JSDoc type cast parens.** `/** @type {T} */ (expr)` is a TypeScript type **assertion** (cast): the parentheses are required, and `/** @type {T} */ expr` (no parens) is *not* a cast. tsv therefore **preserves** the parens of a JSDoc cast everywhere â€” the parser records the cast in an internal `JsdocCast` wrapper node, keyed on the exact paren extent, so `(a.b)` (cast the member) and `(a).b` (cast the base) stay distinct and nested casts keep each level. The public AST stays paren-free, matching acorn/Svelte (which carry no `ParenthesizedExpression`). tsv never *adds* parens â€” only a `@type`/`@satisfies` block comment immediately before a `(` triggers preservation; a non-`@type` comment, or one not adjacent to a `(`, is stripped as ordinary grouping.

The divergence is **JS-vs-TS**, keyed on prettier's parser backend rather than on file type:

- **JS contexts** (`.js`, `.svelte` plain `<script>` â€” prettier's babel parser) **preserve** the cast parens too: prettier's `is-type-cast-comment.js` + the `postprocess` `shouldKeepParenthesizedExpression` check keep a `ParenthesizedExpression` whose immediately-preceding comment is a type cast. tsv **matches** here â€” see the match fixtures [jsdoc_type_cast_svelte](../tests/fixtures/typescript/syntax/comments/jsdoc_type_cast_svelte/), [jsdoc_type_cast_extent](../tests/fixtures/typescript/syntax/comments/jsdoc_type_cast_extent/), [jsdoc_type_cast_nested](../tests/fixtures/typescript/syntax/comments/jsdoc_type_cast_nested/), and [jsdoc_type_cast_keyword](../tests/fixtures/typescript/syntax/comments/jsdoc_type_cast_keyword/). Because tsv formats context-free, these JS matches also **oracle-anchor the preserved layout** that the TS divergence shares â€” the wide cases ([jsdoc_cast_call_arg_long](../tests/fixtures/typescript/expressions/calls/jsdoc_cast_call_arg_long/), [jsdoc_cast_multiarg_long](../tests/fixtures/typescript/expressions/calls/jsdoc_cast_multiarg_long/), [jsdoc_cast_arg_long](../tests/fixtures/typescript/expressions/calls/jsdoc_cast_arg_long/), [jsdoc_cast_paren_long](../tests/fixtures/typescript/calls/chained/jsdoc_cast_paren_long/), [jsdoc_cast_before_eq](../tests/fixtures/typescript/declarations/variable/jsdoc_cast_before_eq/), [jsdoc_cast_paren_span](../tests/fixtures/typescript/declarations/variable/jsdoc_cast_paren_span/), [arrow_jsdoc_cast_body_long](../tests/fixtures/typescript/calls/arrow_jsdoc_cast_body_long/)) pin how the preserved parens break against prettier-plugin-svelte rather than against tsv's own output.
- **TS contexts** (`.ts`, `.svelte` `<script lang="ts">` â€” prettier's oxc-ts parser) **strip** the parens: oxc-ts is cast-unaware (the preservation is gated behind `if (!isOxcTs)`), so prettier silently drops the assertion. tsv preserves â†’ **divergence** (the fixtures listed above). When the inner expression needs parens anyway (e.g. an assignment in `return`), prettier keeps them in both backends, so that case stays a match.

tsv runs one context-free TypeScript formatter, so "preserve" is uniform: it matches prettier in JS contexts and diverges from prettier-TS's strip. This is the **more correct** behavior, not merely a taste call â€” the parens carry meaning, and prettier's own babel path special-cases preserving them (it strips meaningless parens aggressively everywhere else). The cast paren is **opaque to layout heuristics** (expand-last etc.), mirroring acorn's `ParenthesizedExpression`, so a long cast breaks via standard arg breaking and the inner expands inside the preserved parens. An object/array literal inner **hugs** the cast parens (`({â€¦})` / `([â€¦])`) â€” see [jsdoc_type_cast_object](../tests/fixtures/typescript/syntax/comments/jsdoc_type_cast_object/) â€” while a line comment between the `(` and the inner forces a hardline so it can't swallow the inner â€” see [jsdoc_type_cast_gap_comment](../tests/fixtures/typescript/syntax/comments/jsdoc_type_cast_gap_comment/). Negatives that pin the boundary: [jsdoc_type_cast_member](../tests/fixtures/typescript/syntax/comments/jsdoc_type_cast_member/) and [jsdoc_type_cast_nonadjacent](../tests/fixtures/typescript/syntax/comments/jsdoc_type_cast_nonadjacent/) (comments that are not casts), [jsdoc_type_cast_word_boundary](../tests/fixtures/typescript/syntax/comments/jsdoc_type_cast_word_boundary/) (the ASCII `\b` after `@type`/`@satisfies`, mirroring prettier's regex), and [jsdoc_type_cast_ts](../tests/fixtures/typescript/syntax/comments/jsdoc_type_cast_ts/) (never-add + no double-wrap in standalone TS).

**Svelte template / directive positions are not covered by the JS-vs-TS split.** The "JS preserves" rule above holds only inside `<script>` (prettier's babel/oxc path). In a Svelte **template** â€” attribute values (`title={â€¦}`), block tests (`{#if â€¦}`), and mustache tags (`{â€¦}`, `{@html â€¦}`) â€” prettier-plugin-svelte routes the expression through a path that **strips** the cast even in a plain (JS) component. tsv preserves uniformly â†’ divergence, pinned by [jsdoc_cast_template](../tests/fixtures/svelte/syntax/comments/jsdoc_cast_template_svelte_prettier_divergence/) (also a `_svelte_divergence` â€” the cast comment's `leadingComments` attachment differs from Svelte; see [conformance_svelte.md](./conformance_svelte.md) Â§Comment Attachment Differences). The `{@const}` case is a prettier **bug**: `{@const y = /** @type {T} */ (z)}` makes prettier-plugin-svelte emit invalid output `(z}` (it drops the `)`) and then throw on its own output; tsv preserves it correctly, so it is documented here but cannot be pinned as an `output_prettier.*` oracle.

#### Comment normalization (stable quirks)

Prettier has multiple stable forms for comment positioning. tsv normalizes to a single canonical form.

- Computed access comment â€” [trailing_member_computed_comment](../tests/fixtures/typescript/expressions/calls/chained/trailing_member_computed_comment_prettier_divergence/)
- Member access after-call comment â€” [trailing_member_after_call_comment](../tests/fixtures/typescript/expressions/calls/chained/trailing_member_after_call_comment_prettier_divergence/)
- Type-alias union value before-`;` line comment (2-pass: spurious union break, then collapse) â€” [type_alias_value_trailing_comment](../tests/fixtures/typescript/types/comments/type_alias_value_trailing_comment_prettier_divergence/)
- Sequence operand edge comment â€” [operand_edge_comment](../tests/fixtures/typescript/expressions/sequence/operand_edge_comment_prettier_divergence/)
- Value-position parenthesized-assignment trailing line comment â†’ Floated out past `;` (`const y = (a = b); // c`); tsv keeps it inside the parens, consistent with the sequence/block cases (prettier itself keeps the *block* case inside) â€” [init_assignment_paren_line_comment](../tests/fixtures/typescript/statements/variable/init_assignment_paren_line_comment_prettier_divergence/)
- Block comment mid-chain â€” [block_comment_chain](../tests/fixtures/typescript/expressions/calls/chained/block_comment_chain_prettier_divergence/)
- Property signature leading line comment â€” [annotation_simple](../tests/fixtures/typescript/types/comments/annotation_simple_prettier_divergence/)
- Property signature leading block â€” [annotation_leading_block](../tests/fixtures/typescript/types/comments/annotation_leading_block_prettier_divergence/)
- Param/destructuring default block, own-line value â†’ tsv collapses to after `=` (`a = /* c */ b`), prettier to before `=` (`a /* c */ = b`) â€” [param_default_own_line_block_comment](../tests/fixtures/typescript/declarations/function/param_default_own_line_block_comment_prettier_divergence/)

**Computed access comment**: Prettier requires 2 passes to stabilize line comments before computed access (`[0]`). The intermediate form places the comment inside brackets (`[// comment\n0]`), which then normalizes to end-of-line (`[0]; // comment`). tsv reaches the stable form in one pass.

**Sequence operand edge comment**: a redundantly-parenthesized sequence operand carrying a comment on its outer edge â€” leading on the first operand (`fn(((/* c */ x), y))`) or trailing on the last (`fn((x, (y /* d */)))`) â€” has the comment floated out of the sequence parens, matching prettier's fixed point (`fn(/* c */ (x, y))`, `fn((x, y) /* d */)`). Prettier reaches the same forms but is non-idempotent getting there â€” two passes (the comment stays inline on pass 1, floats on pass 2) â€” while tsv reaches the fixed point in one pass; the user's paren form is documented as `unformatted_ours_paren` paired with prettier's first-pass `prettier_intermediate_paren`. Each floated comment keeps its source line-treatment (own-line â†’ own line via hardline, inline â†’ inline via space; the trailing one defers via `line_suffix` past the enclosing comma so it re-parses to the same place), which is what makes the one-pass float idempotent even when the sequence is nested inside other comments. In statement context the trailing edge instead lands before the `;` â€” a genuine divergence; see [Sequence last-operand trailing edge](#comment-relocation) / `sequence/operand_edge_comment_stmt`. Interior operand comments (between two operands) stay inline and match prettier â€” see the regular fixture `sequence/operand_comments`.

**Block comment mid-chain**: When nested grouping parens with block comments are stripped on a member chain (e.g., `/* outer */ (/* inner */ (a).b).c(fn)`), Prettier repositions the inner comment mid-chain (`a /* inner */.b`) and breaks the chain. Prettier requires 2 passes to stabilize the spacing: pass 1 produces `a/* inner */ .b`, pass 2 produces `a /* inner */.b`. Both passes break the chain at the same point. Not JSDoc-specific â€” any block comment before stripped grouping parens triggers this. tsv normalizes directly to the stable form in one pass.

**Property signature leading block**: A block comment between `:` and the type in a property signature has two intentional stable positions (`a: /* block */ X;` after `:`, and `a /* block */: X;` before `:`); both formatters preserve each when given as input. The divergence is in normalizing **unstable** layouts â€” when the user breaks the line around the comment (`a: /* block */\n X;` or `a:\n /* block */\n X;`), tsv compacts to the inline form after `:`, while prettier eventually relocates the block before the `:` (sometimes via a multi-pass convergence). Neither choice is information-destructive; this is purely about which canonical target to favor for ambiguous inputs.

**Param/destructuring default block, own-line value**: The exact analog of the property-signature case at a parameter (or destructuring binding) `=` default. A block comment in the `=`â†’value gap has two intentional inline positions â€” `a = /* block */ b` after `=` (on the value) and `a /* block */ = b` before `=` (on the name) â€” and both formatters preserve each when given as input. When the value is authored on its own line (`a = /* block */\n b`), tsv collapses to the after-`=` form while prettier collapses to the before-`=` form. Block comments inline losslessly, so neither wraps; the only difference is which side of `=` the comment lands on for the ambiguous own-line input.

**Property signature leading line comment**: For a line comment between `:` and an inline-renderable type in a property signature (`{ prop: // c\n X }` â€” covers identifiers, optional `?:`, readonly, computed keys, generics like `Array<X>`, tuples, function types, `typeof`, etc.), Prettier moves the comment past the implicit `;` to end-of-line (`prop: X; // c`); tsv keeps the comment after `:` and drops the type to a continuation line indented one level (`prop: // c\n\t X;`, the [Uniform Forced-Continuation Indent](#uniform-forced-continuation-indent)). Both forms are stable under their own formatter. A multi-member **union** in the same position **diverges** â€” prettier drops the comment onto its own line after `:` while tsv keeps it trailing the `:` + indents the union (the [annotation](../tests/fixtures/typescript/types/comments/annotation_prettier_divergence/) fixture); a multi-member **intersection** likewise **diverges** (prettier keeps it flush, tsv indents â€” see [annotation_continuation_indent](../tests/fixtures/typescript/types/comments/annotation_continuation_indent_prettier_divergence/)). Notably, prettier's end-of-line motion is information-destructive when more than one comment touches the property: leading line + trailing line collapses to `f: X; // leading // trailing` (second `//` becomes text inside the first comment); two leading lines merge **and reverse** order (`g: // c1\n // c2\n X;` â†’ `g: // c2 // c1\n X;`); leading line + trailing block reorders to `h: X; /* trailing */ // leading`. tsv preserves each comment at its authored position as a separate comment node. The end-of-line **relocation** is property-signature-only â€” prettier keeps variable declarations (`const e: // c\n X = ...`) and class properties (`class C { prop: // c\n X }`) in place â€” but tsv's continuation **indent** is universal across all these contexts, so those keep-in-place cases become an indent-only divergence too (the same [annotation_continuation_indent](../tests/fixtures/typescript/types/comments/annotation_continuation_indent_prettier_divergence/) fixture).

### Format-ignore directive

A comment can suppress formatting of the construct that follows it. tsv honors its own tool-neutral `format-ignore` family â€” `<!-- format-ignore -->`, `// format-ignore`, `/* format-ignore */`, and the range markers `format-ignore-start` / `format-ignore-end` â€” **in addition to** prettier's `prettier-ignore` family, which tsv keeps for drop-in compatibility (corpus files use it). Recognition is centralized in `tsv_lang::is_format_ignore_directive` and the two range predicates, shared across the TypeScript, CSS, and Svelte printers.

The `prettier-ignore` family matches prettier exactly (both emit the construct raw), so it needs no divergence fixture of its own. The `format-ignore` family is tsv-native: prettier doesn't recognize it, so prettier reformats the construct while tsv preserves it â€” that difference is the divergence. Most fixtures pair the spellings in one input: a `prettier-ignore`d construct (preserved by both tools, so unchanged in `output_prettier`) sits beside a `format-ignore`d one (reformatted only by prettier), making the `format-ignore` construct the sole divergence and doubling as a drop-in-compat check. The `basic` (template node) and `js_css` (embedded `<script>` + `<style>`) Svelte fixtures carry this control, as do both standalone fixtures.
